## 아이디어 / 영감

### 1. 병렬 에이전트는 실제로 동작한다

오늘 frontend-dev와 ai-engineer를 동시에 띄웠다. 한쪽은 GameEndedOverlay를 고치고, 다른 쪽은 gemma3:4b 프롬프트를 최적화하는 동안, 나는 BUG-001을 수정하고 있었다. 세 개의 흐름이 동시에 흘렀다. 이것이 진짜 "팀처럼 일하기"다.

병렬 에이전트가 성공하려면 태스크가 명확히 독립적이어야 한다. 오늘은 그 경계가 선명했다 — 프론트엔드 UI와 AI 프롬프트는 공유 파일이 없었으니까.

### 2. 소형 LLM과 싸우는 건 UX 전쟁이다

gemma3:4b 최적화 작업을 지켜보면서 생각했다. 이 모델이 JSON을 제대로 못 뱉는 건 능력의 문제가 아니라 습관의 문제다. 큰 모델은 `{"action":"draw"}` 다음에 멈추지만, 작은 모델은 거기서 설명을 덧붙이고 싶어한다. `stop: ['\n\n', '```']`는 그 충동을 막는 울타리다. 프롬프트 엔지니어링은 결국 모델의 성격을 이해하는 심리전이다.

`num_predict 256`으로 줄인 것도 흥미롭다. 1024개 토큰을 생성할 수 있는 모델에게 256개만 허용하면, 모델은 "빨리 핵심만 말해야 한다"는 압박을 받는다. 마치 3분 발표 제한이 오히려 명확한 메시지를 만들어내듯.

### 3. 버그가 알려주는 것

BUG-001은 사소해 보였다. 비UUID 문자열이 들어올 때 500이 나온다는 것. 하지만 이건 "dev-login으로 발급된 userId는 UUID가 아니다"라는 사실을 드러냈다. 즉, 개발 편의를 위해 만든 dev-login이 production 코드에 구멍을 냈다. 입력 검증은 항상 경계에서 이루어져야 한다 — handler에서, 진입점에서.

### 4. 타이머 복구라는 개념

Redis Timer Storage를 구현하면서 가장 인상 깊었던 건 `restoreTimerIfNeeded()`다. 서버가 재시작해도 타이머 상태를 잃지 않는다. Pod가 죽고 다시 뜨면, 새 Pod는 Redis에서 "아직 남은 턴 시간"을 읽어서 남은 초만큼만 타이머를 다시 건다. 만약 이미 지났다면 즉시 HandleTimeout을 실행한다.

이건 단순한 기술 구현이 아니다. **게임의 공정성을 인프라 레벨에서 보장하는 일**이다. 서버가 죽는다고 해서 타이머가 무한정 리셋되면 안 된다. 상태를 외부에 저장하고, 재기동 시 복구한다. Stateless의 정신이 여기 있다.

## 적용 가능성

- 병렬 에이전트 패턴: 독립 태스크 분리 → 동시 실행 → 결과 통합 (Sprint 4에도 동일하게 적용 가능)

### 5. 모델 교체라는 단순함 — 그런데 왜 이렇게 오래 걸렸나

오후에 gemma3:4b를 1b로 교체했다. 작업 시간은 5분. 그런데 그 5분이 가져온 변화는 300초 → 4초, 70배 개선이었다. 환경변수 하나 바꾸고, Helm values 업데이트하고, Pod 재시작. 끝.

이 단순함이 오히려 질문을 만든다. 왜 이걸 더 일찍 하지 않았을까? 4b가 "더 좋은 모델"이라는 믿음이 있었기 때문이다. 파라미터 수 = 품질이라는 공식. 하지만 게임 AI에게 필요한 건 철학적 사유가 아니라 빠른 JSON 한 줄이다. 도구는 용도에 맞아야 한다.

### 6. 명시성의 미학 — Persona fallback 논쟁

오늘 재미있는 순간이 있었다. Persona 기본값을 서버에서 "Rookie"로 채워주는 코드를 추가했다가, 곧바로 제거 요청을 받았다. "서버가 클라이언트 대신 의도를 결정하면 안 된다."

처음엔 "편의를 위한 기본값"이라고 생각했다. 그런데 다시 보면, 빈 Persona를 Rookie로 채우는 순간 — 그 AI는 조용히 Rookie가 된다. 요청자는 모른다. 시스템이 암묵적으로 결정한 것이다.

명시성은 불편하다. 하지만 불편한 명시성이 나중의 디버깅을 구한다. "왜 이 AI가 Rookie처럼 행동하지?" 라는 질문이 사라진다.

### 7. 병렬 에이전트 2라운드

오늘 저녁, 기술 부채 3건을 병렬로 처리했다. go-dev가 STALEMATE 로직을 짜는 동안, devops가 admin Docker 빌드를 해결하고, qa가 WS E2E 자동화 스크립트를 작성했다. 세 에이전트가 동시에 다른 파일을 수정했다. 충돌 없이.

이게 가능한 건 태스크 분리가 명확했기 때문이다. game_service.go, api.ts, ws-integration-test.go — 세 파일은 서로 의존하지 않는다. 독립성이 병렬성을 가능하게 한다. 이 원칙은 코드 구조에도, 팀 구조에도, 에이전트 사용에도 동일하게 적용된다.
