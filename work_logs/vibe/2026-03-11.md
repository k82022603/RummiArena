# 바이브 로그

- **날짜**: 2026-03-11
- **주제**: 뷰 분리 전략, 폴리글랏 백엔드

## 아이디어 / 영감

- **실시간 vs 복기 뷰 분리**: 같은 게임 데이터를 다른 관점으로 보여주는 것이 핵심. 실시간은 1인칭(정보 제한), 복기는 4분할(전체 공개). 하나의 데이터, 두 개의 렌즈.
- **스냅샷 기반 복기**: 매 턴 DB 스냅샷만 쌓으면 복기 기능이 자연스럽게 나온다. 이벤트 소싱까지 갈 필요 없이 턴 단위 스냅샷으로 충분.
- **폴리글랏은 학습 투자**: Go를 game-server에 쓰는 건 성능보다 "Go 실전 경험"이 진짜 이유. 프로젝트 목적이 학습이니까 이게 맞다.

## 왜 떠올랐는지
- 시뮬레이션 3종을 만들면서 관전→4분할→1인칭으로 뷰가 진화하는 과정에서, "실시간은 1인칭이 맞고 4분할은 복기에 쓰면 완벽하다"는 깨달음
- 백엔드 기술 논의에서 "Go+NestJS가 가능한가?"라는 질문이 폴리글랏 결정의 시작점이었음

## 적용 가능성
- 복기 뷰는 Phase 4 Sprint 6에서 구현 예정 (FR-009)
- 복기 데이터는 AI 전략 비교 분석 리포트의 원천 데이터가 됨
- Go game-server 경험은 K8s 네이티브 생태계 이해에 직접 기여

## 메모
- 시뮬레이션 HTML은 실제 구현 시 UI 프로토타입으로 재활용 가능
- "하나의 데이터, 두 개의 렌즈" 패턴은 관리자 대시보드에도 적용할 수 있음 (운영 관점 뷰)
