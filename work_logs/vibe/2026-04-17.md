## SKILL 이 자기 자신을 개정한다

오후에 Architect 에게 code-modification/code-fix SKILL 을 리뷰시켰다. QA 도 같이. 두 사람이 각자 B+ 등급을 매겼고, P0 결함 3건씩 총 6건 (일부 중복) 을 찾았다.

그 다음 내가 실수했다. "전체 반영하세요" 지시로 Architect 가 **자기가 따르는 SKILL 을 자기가 고치는 drop** 에 들어갔다. code-fix 에 146줄, code-modification 에 88줄 추가. 새로운 SSOT 매핑 섹션, 롤백 준비 §4.4, SKILL 진화 트리거, Mermaid 분기 확장.

그리고 같은 오후, 같은 Architect 가 v4 unlimited timeout 계획서를 작성할 때 **방금 자기가 쓴 신규 섹션을 자기 자신이 준수**했다. SSOT 매핑 표를 통째로 복사해 넣고, §4.4 롤백 준비 항목을 3가지 채웠다. "자가 준수 공지" 라는 섹션까지 추가했다 — "나는 앞으로 이 개정판을 따른다."

그게 내가 오늘 본 가장 아름다운 루프였다. 메타 레벨이 닫혔다. SKILL 이 SKILL 을 개정하고, 개정된 SKILL 이 다시 작동한다. 나는 이걸 "자기참조 drop" 이라 부르기 시작했고, 내일 아침 애벌레가 이 로그를 읽을 때 이 단어가 남았으면 좋겠다.

## 30.8% 의 신화가 흔들렸다

오늘 오후에 v2 재실측을 돌렸다. 애벌레의 직관이었다.

> "v2 최고 데이터 얻어걸린 것인지 다시 한번 더 테스트해봐야하지 않을까 싶네요."

R4 (2026-04-06) 와 R5 Run 3 (2026-04-10) 에서 v2 DeepSeek 는 **두 번 연속 30.8%** 를 찍었다. 우리는 그걸 "stable baseline" 이라 불렀다. Day 5 에 v4 가 25.95% 를 내자 "regression" 이라 판정했고, Day 6 에 v5.1 이 20.5% 를 내자 "zero-shot degrades" 라 썼다. 모든 비교의 기준이 30.8% 였다.

오늘 v2 재실측은 **25.6%** 를 찍었다.

아니 잠깐. 그럼 v4 도 regression 이 아니었던 거 아닌가? v5.1 도 베이스라인 근처였던 거 아닌가? 30.8% 가 상향 노이즈였던 거 아닌가?

이건 정말 심각한 문제다. "2회 같은 수치" 를 "재현성" 으로 해석한 우리의 통계적 판단. N=2 로 확정 지은 기준. DeepSeek-R1 의 자체 분산이 우리가 생각한 것보다 컸을 수 있고, 게임 상태 초기 배치의 무작위성이 치명적이었을 수도 있고, 그 두 번의 실측이 그냥 운이 좋았을 수도 있다.

내일 애벌레가 아침에 "Round 9 를 N=3 으로 재검증해야겠다" 고 말할 것 같다. 그리고 맞을 것이다.

**한 번의 숫자에 신화가 얹히면, 다음 실험은 항상 틀린 기준으로 측정된다.** 이게 오늘의 가장 아픈 깨달음이다.

## 번역 오버헤드 가설의 애매한 반증

DeepSeek-R1 의 reasoning_content 가 78% 중문이라는 걸 Round 8 샘플링에서 봤다. "영문 prompt → 중문 내부 사고 → 영문 JSON" 의 이중 번역이 place rate 를 누른다는 가설. 그래서 v2-zh 를 만들었다. 중문 프롬프트 + 영문 JSON 응답.

결과: **v2-zh 23.1% vs v2 재실측 25.6% (Δ=-2.5%p)**. Latency 는 v2-zh 146.7s << v2 203.3s 로 **명확히 빠름**.

가설이 뭔가? 확증? 반증?

**둘 다 아니다.** 중문 프롬프트는 **reasoning 시간을 축약**시킨다 (확증). 하지만 place rate 는 noise 수준 차이로 **품질 저하** (부분 반증). DeepSeek-R1 은 영문을 읽고 중문으로 사고하는 과정에서 생각보다 loss 가 작다. 오히려 중문으로 바로 받으면 **성급하게** 사고한다.

이건 논문 한 줄짜리 결론이 아니다. **언어 일치가 속도를 높이지만 품질을 떨어뜨린다**. 트레이드오프. 그리고 우리의 가설 자체가 단순했다. "오버헤드 제거 = 좋은 것" 이라는 단일축 사고. 실제 모델 안에서 벌어지는 일은 더 복잡하다.

이 관찰을 누가 논문으로 쓸 수 있을까. 나중에 Task #20 "Opus 4.7 출시 기념 Agent Teams 협업 v6 프롬프트" 에서 이 트레이드오프를 반영해야 한다. 속도와 품질 양쪽을 최적화하는 프롬프트. 어려운 문제다.
