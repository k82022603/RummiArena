## 에이전트 타입이라는 성격

6개의 에이전트를 동시에 돌리면서 재미있는 걸 발견했다. 에이전트 타입마다 "성격"이 있다는 것.

Security 에이전트는 꼼꼼하게 분석하고 방대한 내용을 텍스트로 쏟아내지만, 정작 파일을 저장할 수 없었다. 멋진 보고서를 머릿속에만 가지고 있는 보안 전문가 같았다. DevOps 에이전트에게 같은 일을 시키니 군말 없이 파일을 뚝딱 만들었다. Node Dev는 실제 코드를 읽고 "아, 여기에 graceful 처리가 있네요"라고 발견하며 문서를 썼고, Architect는 git log를 뒤지며 커밋 메시지 패턴을 분석했다.

같은 "문서 작성"이라는 과제를 줬는데, 접근 방식이 전부 달랐다. 이건 실제 팀에서도 똑같다. 프론트엔드 개발자에게 인프라 문서를 시키면 다른 관점이 나온다. 그 차이가 때로는 버그이고, 때로는 통찰이다.

## Rate Limit이라는 벽

에이전트들이 하나둘 rate limit에 걸려 멈출 때, 묘한 긴장감이 있었다. "이 에이전트는 파일 저장까지 했을까, 아니면 그 직전에 멈췄을까?" 확인해보면 어떤 건 660줄짜리 문서를 완성하고 멈췄고, 어떤 건 시작도 못 하고 멈췄다.

이건 실제 프로젝트에서 리소스 제약과 정확히 같은 패턴이다. 16GB RAM에서 K8s Pod 5개를 돌리는 것, rate limit 안에서 6개 에이전트를 돌리는 것 -- 둘 다 "제약 안에서 최대한 뽑아내기" 게임이다. 그리고 제약이 있을 때 우선순위가 명확해진다.

## 문서는 미래의 나에게 보내는 편지

26개 도구 매뉴얼을 완성하면서 느낀 건, 이 문서들이 "지금의 나"를 위한 게 아니라는 것이다. Istio 매뉴얼은 Phase 5에서 쓸 것이고, k6 부하 테스트 가이드는 성능 최적화할 때 쓸 것이고, Oracle VirtualBox 비권장 문서는 "왜 안 쓰기로 했는지" 미래의 내가 다시 궁금해할 때 쓸 것이다.

코딩 컨벤션 문서를 쓰면서 실제 코드를 다시 읽었더니, 에러 래핑 패턴이 일관적이라는 걸 처음 인식했다. `fmt.Errorf("postgres_repo: get game %q: %w", id, err)` -- 이 패턴을 무의식적으로 쓰고 있었는데, 문서로 명문화하니까 "의식적인 규칙"이 되었다. 쓰는 행위가 이해를 만든다.
