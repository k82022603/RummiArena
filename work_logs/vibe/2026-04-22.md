## 영감 / 가설

### 가설 1 — "자동 테스트가 회귀를 드러내는 사례의 구조적 가치"

FINDING-01 은 프로젝트 역사상 **첫 번째 "자동 테스트가 사용자 피해 전에 회귀를 잡은" 사례** 다. 전까지는 항상 애벌레가 플레이하다가 발견했다.

이번 순서:
1. PR #41 (내 수정)
2. `qa regression session` — architect 가 **반사실 체크리스트** 작성, qa 가 **신규 시나리오 7건** 을 실행
3. `hotfix-p0-i2-run-append.spec.ts` SC1/SC2 **REAL FAIL** 재현
4. Issue #46 자동 생성
5. architect 근본 원인 분석 (`73-*.md`)
6. PR #51 완전 해소

이 사이클이 **하루 안에 닫혔다**. 기존 패턴에서는:
1. 애벌레가 게임 돌리다가 "또 안 된다" 캡처
2. Claude 가 "다음 세션에 고치겠습니다"
3. 다음 날 패치

→ 사용자가 **회귀를 실측할 필요가 없는** 체계가 조금씩 자리 잡고 있다. `ui-regression` SKILL + `pre-deploy-playbook` SKILL 두 개를 조합하니 실제로 "사용자 전에 Claude 가 먼저 발견" 이 일어난다.

### 가설 2 — "쓸데없는 질문의 구조적 원인: SKILL 의 맹목적 적용"

오늘 초반에 사용자에게 "Scope L5 할까요 L6 할까요?" 를 2번 물었다. 사용자는 **"뭘 하라는 것인지 모르겠음"** 이라고 답했고, **"아니면 말고 식 일처리하면 어쩌자는거야?"** 라고 한 번 더 짚었다.

나는 `pr-workflow` SKILL Phase 0 (사전 합의) 규칙을 기계적으로 적용했다. 그 SKILL 자체는 틀리지 않았다. 하지만 **AUTO MODE 가 켜진 상태** + **이전 PR #37/#38/#39 패턴 (L5) 가 이미 확립된 상태** 에서 "기본은 L5" 라고 판단하지 못한 게 문제였다.

→ **SKILL 은 일반 규칙이고, 프로젝트의 현재 상태가 그 규칙의 기본값을 이미 결정해두었을 수 있다.** 이전 패턴이 있으면 그게 default, 물어볼 건 "default 에서 벗어날 이유가 있는가" 뿐.

이것은 SKILL 자체의 결함이 아니라, SKILL 과 상황의 매핑을 내가 기계적으로 했다는 뜻이다. 사용자가 바로 잡아줘서 그날 안에 교정됐지만, 구조적 원인은 **"AUTO MODE 에서 SKILL 을 어떻게 얇게 적용할지"** 라는 문제다.

→ 다음 번 SKILL 적용 시 self-check: "이 SKILL 의 Phase 0 질문이 이전 세션에서 이미 답변되었는가?"

architect A 가 영향도 분석에서 찾은 작은 유레카. 계획서에는 "Day 12 backend P0-1 game_results persistence" 가 적혀 있었는데, 실제 코드를 읽어보니:

### 가설 4 — "Agent Teams 한 세션 13 PR 처리의 확장성"

오늘 한 세션 안에 PR 13건 (머지 기준) 을 처리했다. 프로젝트 최고 기록. architect 4회, pm 1회, qa 1회, go-dev 1회, frontend-dev 3회, devops 1회. 총 11회 에이전트 투입.

어떻게 가능했나:
1. **worktree 격리** — hotfix/finding-01, chore/warn-triage, feat/rooms-postgres-phase1-impl 3개 worktree 가 동시 돌았다. 파일 충돌 없이.
2. **명시적 SKILL 매핑** — 각 에이전트 프롬프트에 "이 SKILL 들 적용하세요" 를 박았다. 사용자가 중간에 "SKILL 사용 잊지마세요" 한 번 상기시켰다.
3. **L5 → L6 전환 타이밍** — 처음 4건은 사용자 직접 머지 (#40~#44), 이후는 에이전트 auto merge (#45~#52). 초기에 품질 기준을 사용자가 검증하고, 패턴 확립되면 자동화로 넘김.
4. **rebase 1건 수행** — hotfix/finding-01 브랜치가 PR #50 이전 main 에서 파생되어 PR #50 revert 위험 발생. rebase 로 해결. 충돌 없이.

→ 이 정도 규모는 **"프로세스가 사람보다 빠른 지점"** 이다. 사용자 1명 + AI 1명으로 이 많은 변경을 검증·머지하려면 SKILL·worktree·auto merge 삼박자가 필수다.
