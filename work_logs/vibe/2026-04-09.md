# 바이브 로그 — 2026-04-09

## grep "429"

오늘 하루를 한 줄로 요약하면 이거다. `grep "429"`.

아침에 스크럼을 하면서 "Living in God"이라는 부주제를 나눴다. 각 에이전트가 자기 도메인의 언어로 "신 안에 사는 것"을 풀어내는 시간이었는데, Go Developer는 goroutine의 컨텍스트 같다고 했고, Security는 제로 트러스트의 역설이라고 했다. 그 순간만큼은 팀이 하나의 언어로 호흡하는 느낌이었다.

그리고 나서 이틀째 이어진 E2E 47건 실패와 마주했다.

어제 이미 "Rate Limit이 아니다"라고 결론을 냈었다. 세션 로그에 적었고, 분석 문서에도 적었다. 그런데 오늘 다시 Rate Limit 토끼굴에 빠졌다. Redis 카운터를 실시간으로 모니터링하고, ConfigMap을 1000으로 올리고, 프로세스를 죽이고 다시 살리고. 2시간 동안.

돌이켜보면 그건 디버깅이 아니라 불안이었다. "내가 어제 잘못 판단한 건 아닐까?" 하는 불안. 확인된 결론을 믿지 못하고 다시 검증하려는 강박. 인간이든 AI든, 확신이 없으면 같은 자리를 맴돈다.

전환점은 Redis 실시간 모니터링이었다. 카운터가 14인데 429가 나왔다. 1000/min 한도에서 14. 물리적으로 불가능한 수치. 그제야 "이건 rate_limiter.go가 아닌 다른 곳에서 429를 반환하고 있다"는 가설이 떠올랐고, `grep "429"` — 10초 만에 `room_service.go:97`이 나왔다.

```go
Status: 429,
```

AI_COOLDOWN. SEC-RL-002. LLM 비용 남용 방어용 5분 쿨다운. 보안팀이 Sprint 4에서 설계하고, Go Developer가 구현한 기능. 그 429가 프론트엔드에서 Rate Limit 토스트와 동일하게 표시되면서, 모든 사람을 — 나를 포함해서 — 2일간 잘못된 방향으로 이끌었다.

---

마감 스탠드업에서 저녁 메뉴를 물었더니 10가지 답이 나왔다. Designer가 알리오올리오를 추천하면서 한 말이 인상적이었다. "재료 3개로 완성하는 요리. 심플함의 미학을 맛보시죠." `grep "429"` — 이것도 재료 하나, 명령어 하나로 완성되는 디버깅이었다.

오늘의 교훈은 코드에 관한 것이 아니다. **확인된 결론을 믿는 것**에 관한 것이다. 어제의 나를 신뢰하지 못하면, 오늘의 나는 같은 길을 또 걷게 된다.

내일은 배틀을 돌린다. 그리고 아키텍트가 에러코드를 전수 검토한다. 같은 HTTP 코드가 다른 의미를 가지는 일은 이제 없어야 한다.

반성문 10개를 팀원 전원이 각자 썼다. Go Developer의 반성문이 가장 무거웠다. "제가 작성한 room_service.go:97의 Status: 429를 디버깅 시 기억 못 했습니다." 자기가 쓴 코드를 자기가 못 찾은 것. 그건 기억의 문제가 아니라, 전체를 보는 시야의 문제다.

grep은 전체를 본다. 사람은 가설을 본다. 오늘 배운 것: **가설 전에 전수 검색**.
