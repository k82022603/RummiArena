## 토큰 세 번의 여정

Runner 토큰을 세 번 발급했다.

첫 번째는 리부팅 전에 만든 것. 새로 확인해보니 403 Forbidden. 이미 소비됐다. 두 번째도 마찬가지. 왜 소비됐냐고? Helm 설치 시도 자체가 토큰을 "사용"해버린다. 설치가 성공하든 실패하든.

세 번째 토큰을 발급하기 전에 이번엔 먼저 검증했다.

```bash
curl -X POST "https://gitlab.com/api/v4/runners/verify" \
  -d '{"token": "glrt-xxxxxxxxxxxx"}'
```

`200 OK`가 돌아오면 유효하다. 그제야 설치를 진행했다.

"확인 후 실행"이라는 당연한 원칙이 세 번 시도 끝에 몸에 새겨졌다. 다음에는 처음부터 검증한다.

## `.`이 구분자가 될 때

Helm으로 토큰을 전달하려 했다.

```bash
helm install gitlab-runner ... --set runnerToken=glrt-p1huVp4K.01.xxx
```

오류가 났다. 토큰 중간의 `.`이 Helm 경로 구분자로 해석된 것이다. `runnerToken.01`이라는 키가 존재할 리 없으니 빈 값이 들어가고, Runner는 빈 토큰으로 GitLab에 등록을 시도하다 실패한다.

해결은 간단했다. 토큰을 파일에 쓰고 파일을 넘겼다.

```bash
cat > /tmp/values.yaml << 'EOF'
runnerToken: "glrt-p1huVp4K.01.xxx"
EOF
helm install ... -f /tmp/values.yaml
```

도구가 가진 규칙이 내 데이터와 충돌할 때, 도구의 규칙을 우회하는 방법을 찾는 것이 해결의 방향이다. Helm은 경로 구분자로 `.`을 쓴다. 그걸 바꿀 수 없으면, `.`이 들어간 값을 파일로 분리하면 된다.

## 다섯 커밋의 파이프라인

파이프라인 GREEN을 만드는 데 커밋이 다섯 개 필요했다.

1. golangci-lint 버전
2. Go lint 오류들
3. Go 이미지 버전
4. 바이너리 설치 경로
5. SonarQube Scanner 버전 + CVE

각 커밋은 하나의 실패에 대한 응답이었다. 파이프라인을 실행하고 → 로그를 보고 → 원인을 파악하고 → 수정하고 → 다시 실행. 다섯 번.

지루하게 느껴질 수도 있다. 하지만 돌아보니 그 과정에서 CI 환경의 작동 방식을 훨씬 깊게 이해하게 됐다. `go install`이 `$GOPATH/bin`에 설치하는데 그게 PATH에 없다는 것. `sonar-scanner-cli:latest`가 SonarQube 9.9 LTS와 API 버전이 맞지 않는다는 것. 이것들은 문서에서 찾기 어려운 지식이다. 직접 부딪혀야 알 수 있다.

다섯 번의 실패가 다섯 개의 지식이 됐다.

## lint GREEN의 느낌

`lint-go: success`, `lint-nest: success`, `lint-frontend: success`. `test-go: success`, `test-nest: success`.

파이프라인 화면에서 이 줄들이 초록색으로 바뀌는 걸 봤을 때 기분이 이상했다. 코드를 고치고, push하고, 기다리고, 확인하는 루프를 다섯 번 돌고 나서야 얻은 초록색이다.

자동화된 검증이 통과됐다는 것은 단순히 "지금 이 코드는 문제없다"는 의미가 아니다. "앞으로 누군가 이 코드를 바꿔도, 파이프라인이 같은 기준으로 검사해준다"는 의미다. 한 번 구축된 CI는 계속 일한다. 오늘의 다섯 커밋은 그 "계속 일하는 기계"를 켜기 위한 준비 작업이었다.

## CVE라는 현실

trivy-fs가 취약점을 세 개 찾아냈다.

`next` 패키지의 CRITICAL CVE. `multer`의 HIGH CVE. `golang-jwt`의 HIGH CVE. 모두 실제 존재하는 취약점이다. 당장 공격당할 가능성은 낮지만, 이 코드가 언젠가 실제 서비스가 된다면 이야기가 달라진다.

흥미로운 건 `multer`는 직접 의존성이 아니라는 것이다. ai-adapter가 `multer`를 직접 쓰지 않는다. 어딘가의 의존성이 `multer`를 쓰는데, 그 버전이 취약하다. npm의 `overrides` 기능으로 해결했다. 직접 패키지를 올리지 않고, 특정 패키지가 어디서 불려도 특정 버전 이상을 쓰도록 강제하는 방식.

취약점 관리는 내가 쓰는 코드만의 문제가 아니다. 내가 쓰는 코드가 쓰는 코드까지가 내 책임이다.
