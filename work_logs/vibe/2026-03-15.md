# 바이브 로그 -- 2026-03-15

## 동시 push라는 실용주의

오늘 흥미로운 설계 결정을 하나 했다. GitHub와 GitLab 두 원격에 동시 push하는 것.

처음엔 이게 이상하게 느껴졌다. 코드를 두 곳에? 그런데 가만히 생각해보니 이건 실용주의의 전형이다. GitHub는 협업과 이슈 추적의 허브고, GitLab은 CI/CD 파이프라인 엔진이다. 두 도구가 각자의 역할에서 최선이다. GitLab 미러링은 유료다. 그러면 `git remote set-url --add --push`로 두 원격을 push URL에 등록하면 된다. 단일 `git push origin main` 한 번으로 양쪽이 동시에 업데이트된다.

도구를 억지로 하나로 통일할 필요가 없다. 잘하는 걸 맡기면 된다.

## 함정의 미학

`glab auth login`을 그냥 실행하면 github.com에 연결된다. GitLab CLI인데.

처음엔 버그라고 생각했다. 하지만 조금 더 생각하니 이건 UX 설계 문제다. 기본값이 잘못 설정된 것인지, 아니면 glab이 git remote를 파싱해서 GitHub remote가 있으면 GitHub를 기본으로 사용하는 건지. 어쨌든 `--hostname gitlab.com`이라는 한 줄이 없으면 전혀 다른 곳에 로그인된다.

이런 함정들이 쌓여서 문서가 된다. 내가 겪은 ISS-02는 이제 가이드 3.3절의 경고 박스가 됐다. 실패는 유실되지 않고 다음 사람을 구한다.

## Elasticsearch가 가르쳐준 것

SonarQube가 시작을 거부했다. 이유는 `Xms ≠ Xmx`. JVM 힙의 초기 크기와 최대 크기가 다르면 Elasticsearch는 부팅을 거부한다. 보안상의 이유다 — 런타임에 힙이 늘어나면 메모리 레이아웃이 예측 불가능해지고, 그건 잠재적인 정보 유출 경로가 된다.

이걸 알면서도 `docker restart`로 고치려 했다. restart는 컨테이너를 재시작하지만 환경변수는 docker-compose 파일에서 읽어오므로, 파일을 바꿔도 기존 컨테이너 설정이 그대로다. `down`으로 컨테이너를 완전히 삭제하고 `up`으로 새로 만들어야 한다.

당연한 것도 직접 부딪혀야 기억에 남는다.

## PLACE_TILES 프리뷰라는 신뢰

타일을 드래그해서 보드에 놓아도 즉시 서버에 가지 않는다. 노란 점선 테두리와 반투명으로 "아직 확정 안 됨"을 표시한다. 확정 버튼을 눌러야 진짜 이동.

이 설계 결정의 본질은 신뢰다. 사용자가 실수를 취소할 수 있어야 한다는 것. HTTP 폼에서 Submit 버튼이 있는 이유와 같다. 실시간 게임에서도 사용자에게 "확인할 시간"을 준다.

기술적으로는 `pendingGroupIds: Set<string>`이라는 간단한 상태지만, UX적으로는 "나는 네 실수를 용납한다"는 선언이다.

## 숫자가 말을 거는 순간

SonarQube가 처음 분석 결과를 돌려줬다. 화면에 숫자들이 떴다. bugs=0, vulnerabilities=0. 그런데 code_smells=18.

18개. 처음엔 당황했다. 이미 잘 작동하는 코드인데, 18개의 문제가 있다고? 그런데 목록을 열어보니 달랐다. "이 문자열이 5번 중복됐습니다." "이 함수의 복잡도가 17입니다. 15 이하로 줄이세요."

이건 버그가 아니다. 냄새다. 지금 당장은 문제없지만, 쌓이면 썩는다. 그 차이를 SonarQube는 `bug`와 `code_smell`이라는 단어로 정확히 구분해서 알려준다. 언어가 정밀할수록 생각이 정밀해진다.

18개를 5개로 줄이는 데 한 세션이 걸렸다. CRITICAL 13건. 문자열 상수를 정의하고, 함수를 작게 쪼개고. 작업 자체는 단순했다. 하지만 그 과정에서 `ConfirmTurn` 함수가 왜 그렇게 길었는지 이해하게 됐다. 스냅샷 처리, 랙 계산, 엔진 호출, 결과 처리가 한 함수에 뭉쳐있었다. 하나씩 이름을 붙여 분리하고 나니, 함수가 하나의 문장처럼 읽혔다.

코드를 정리한다는 것은 생각을 정리하는 것이다.

같은 파일, 같은 내용. 그런데 어디서 보느냐에 따라 0%와 28.4%의 차이가 생긴다. 관점이 곧 현실이다.
