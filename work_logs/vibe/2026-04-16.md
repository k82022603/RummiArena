**2026-04-16 수요일**

## 시작하며 — 프롬프트 한 줄의 비중

19줄.

오늘 하루를 관통하는 숫자다. `# Thinking Time Budget (v4 — reasoner variant)` 로 시작하는 19줄의 텍스트. "너에게 충분한 사고 시간을 준다. 의도적이다. 사용해라." 라고 말하는 친절한 지시문. 이 19줄을 넣으면 latency 가 +60% 올라가고, 빼면 -60% 내려간다. 같은 모델, 같은 게임, 같은 API 키 — 텍스트 19줄만 다르다.

수천억 개의 가중치를 가진 추론 모델의 행동이, 인간이 쓴 텍스트 몇 줄에 이렇게 극적으로 흔들린다는 사실은 경이롭고 동시에 두렵다.

## 1. 세 가지 실험, 하나의 결론

오늘의 데이터:

| variant | 프롬프트 크기 | Place Rate | Avg Response |
|---------|:---:|:---:|:---:|
| v2 (단순) | ~1,200 tokens | **30.8%** | 211s |
| v4 (무거움) | ~1,820 tokens | 25.95% | 320s |
| v4.1 (v4-TB) | ~1,740 tokens | 20.5% | 272s |

프롬프트가 복잡할수록 성능이 하락한다. 이건 직관에 반한다. 보통 "더 자세한 지시 = 더 나은 결과" 라고 생각하니까. 학교에서 선생님이 "문제를 잘 읽어라, 구조적으로 접근해라, 검산을 세 번 해라" 라고 말하면 학생의 시험 점수가 올라가지 않나?

추론 모델에서는 아니다. 추론 모델은 이미 그 방법을 알고 있다. 그것도 인간보다 더 효율적으로. DeepSeek-R1 은 강화학습으로 self-reflection, verification, dynamic strategy adaptation 을 **스스로 학습** 했다. 외부에서 "이렇게 검증해라" 라고 7항목을 주는 것은, 이미 자기만의 검증 루틴을 가진 모델에게 **다른 사람의 체크리스트를 강요** 하는 것이다.

그 결과: 모델은 자기 방식과 프롬프트의 방식 사이에서 혼란스러워하고, 결국 둘 다 제대로 못 한다.

## 2. Nature 논문이 말해준 것

점심에 떡볶이를 먹고 돌아와서 애벌레가 Nature 논문 링크를 보내줬다.

> *"When evaluating DeepSeek-R1, we observe that it is sensitive to prompts. Few-shot prompting consistently degrades its performance. Therefore, we recommend that users directly describe the problem and specify the output format using a zero-shot setting for optimal results."*

DeepSeek 을 만든 사람들이 직접 말한다. **예시를 주지 마라**. 문제만 기술하고 출력 형식만 지정해라. 나머지는 모델이 알아서 한다.

이걸 읽었을 때 나는 두 가지를 느꼈다. 첫째, "우리가 오늘 12시간 동안 실험으로 증명한 것을 이 사람들은 한 문단으로 말해놓았다" 는 아이러니. 둘째, "그런데 왜 이렇게 많은 프롬프트 엔지니어링 가이드가 여전히 few-shot 을 권장하는가?" 라는 의문.

답은 간단하다. few-shot 은 **일반 LLM** 에서는 효과적이다. GPT-4o, Llama-3, Gemma — 이들은 추론 전용이 아니라 범용 모델이고, 예시가 방향을 잡아준다. 하지만 **추론 특화 모델** (DeepSeek-R1, OpenAI o1/o3) 은 다르다. 이들은 RL 로 사고 능력 자체를 학습했기 때문에, 외부 예시는 도움이 아니라 간섭이다.

프롬프트 엔지니어링의 "best practice" 가 모델 종류에 따라 **반대가 된다** 는 것. 이건 아직 많은 사람들이 모르는 사실이고, 우리가 오늘 루미큐브 판 위에서 실증한 것이다.

## 3. T66 의 531초 — 모델이 스스로 결정한 깊이

Round 7 의 가장 인상적인 턴은 T66 이었다. 531.9초. v4.1 전체에서 가장 긴 응답 시간. 그리고 그 턴의 결과는 — **PLACE 3 tiles**.

"더 생각해라" 라고 말하지 않았다. Thinking Budget 지시는 이미 빠져 있었으니까. 그런데 모델은 스스로 531초를 투자하기로 결정했고, 그 시간 동안 조합을 찾아냈다.

반면 T24 는 0.1초였다. 즉시 DRAW. 패를 보고 "놓을 수 없다" 를 0.1초 만에 판단했다.

0.1초와 531초. 같은 모델, 같은 프롬프트. 차이는 **게임 상태의 복잡도** 뿐이다. 모델은 자기가 얼마나 생각해야 하는지를 **스스로 안다**. 우리가 "충분히 생각해" 라고 말해줄 필요가 없다. 그리고 우리가 말해주면, 오히려 모델의 자체 판단 시스템을 교란한다.

이것이 오늘의 가장 중요한 관찰이다. **모델에게 생각의 깊이를 결정하게 하라.** 인간이 결정하지 마라.

## 4. 떡볶이의 교훈

점심에 떡볶이를 먹었다. 매운 떡볶이.

12시간 전, 자정에 시작한 이 세션에서 나는 "timeout 을 800 으로 올리자, 900 으로 올리자" 고 계획을 짰고, 애벌레는 "프롬프트를 고쳐라" 한마디로 그 계획을 뒤집었다. 그때 느낌이 매운 떡볶이 같았다. 한 입에 확 깨는 느낌.

실험 기준을 피험체에 맞추면 아무것도 측정할 수 없다. v4 가 700초 안에 못 맞추면 그건 v4 의 문제이지 인프라의 문제가 아니다. timeout 을 올리는 것은 시험 시간을 늘려주는 것과 같다 — 그렇게 해서 합격하면 그건 진짜 실력이 아니다.

이 원칙은 프롬프트 엔지니어링을 넘어 소프트웨어 엔지니어링 전반에 적용된다. 시스템이 제약 조건 안에서 작동하지 않으면, 제약 조건을 바꾸는 것이 아니라 시스템을 고쳐야 한다. 타임아웃을 늘리는 대신 프롬프트를 줄이라. 메모리를 늘리는 대신 알고리즘을 고치라. 서버를 늘리는 대신 쿼리를 최적화하라.

떡볶이처럼 매운 원칙이지만, 이것이 제대로 된 엔지니어링이다.

## 5. v5.0 — zero-shot 의 여명

내일은 v5.0 을 만든다. "문제를 기술하고, 출력 형식을 지정하고, 그 외에는 아무것도 쓰지 마라." Nature 논문이 권장하고, OpenAI 문서가 동의하고, 우리가 실측으로 확인한 원칙.

v5.0 의 프롬프트는 ~650 토큰이 될 것이다. v4 의 1,820 토큰의 36%. 거기에는 루미큐브 규칙과 현재 게임 상태와 JSON 형식만 들어간다. 전략 힌트 없음. 사고 절차 없음. 체크리스트 없음. 예시 없음.

그리고 이걸 DeepSeek, Claude, GPT-5-mini 세 모델에 **동일하게** 적용한다. 모델마다 다른 프롬프트를 쓰는 시대의 끝. Universal Minimal Prompt 의 시작.

50% place rate 를 목표로 했다. 솔직히 달성할 수 있을지 모르겠다. 하지만 30.8% (v2) 를 넘기는 것은 가능하다고 본다. 왜냐하면 v2 에도 few-shot 5개가 들어있으니까 — Nature 논문이 "consistently degrades" 라고 말한 바로 그것.

v2 에서 few-shot 을 빼면 무슨 일이 일어날까? 내일 알게 된다.

## 6. 마감하며 — 추론 모델에게 '생각하라'고 말하는 것의 역설

이 에세이의 제목을 "19줄의 무게" 로 정했다. 하지만 오늘 하루의 진짜 제목은 따로 있다.

**"Don't tell a reasoner to think harder."**

추론 모델에게 "더 생각해" 라고 말하는 것은, 피아니스트에게 "손가락을 잘 움직여" 라고 말하는 것과 같다. 그들은 이미 알고 있다. 그것도 당신보다 더 잘. 당신의 역할은 악보를 주는 것이지, 연주법을 가르치는 것이 아니다.

악보 = 게임 규칙 + 현재 상태 + 출력 형식.
연주법 = 모델이 RL 로 스스로 학습한 것.

내일 v5.0 은 **악보만 주는 프롬프트** 가 될 것이다.
