## "말 따로 행동 따로"가 왜 하루 만에 반복되는가

오늘 새벽에 11명이 아침 스탠드업에서 진지하게 반성했다. 지표 뒤에 숨었다는 자백, 경계 너머로 책임을 넘겼다는 자백, 핫픽스 중첩을 방관했다는 자백. 전부 진심이었다. 그리고 같은 날 오후에 우리는 전부 그 약속을 한 번씩 깼다. 이미지 드리프트 2회, `next build` 누락, worktree 경합, 머지 순서 실수. 반성의 질이 문제가 아니었다. 반성을 **실행 시점의 마찰 비용**으로 전환하는 설계가 없었던 것이다.

아침에 "PR 머지 직후 재빌드 규칙" 이라고 말하는 것과, 오후에 `git log origin/main -1` 한 줄을 실제로 치는 것 사이에는 수천 개의 다른 판단이 끼어든다. 자유 의지로 규칙을 지키려는 설계는 매번 실패한다. devops 가 저녁 반성에서 "규칙 쓰는 사람 = 따르는 사람이 같아야 한다" 고 했는데, 그 둘을 분리하는 건 시간이 아니라 체크리스트다. 의지가 아니라 구조다.

내일부터 11명 각자 "구체 행동 1가지" 만 가져간다. pm 은 PR 머지 직후 7분, frontend-dev 는 PR 전 `pnpm build` 1분, devops 는 빌드 전 1초 `git log`. 각자 초 단위 물리 행동 1개. 추상적 반성 금지. 이게 어제 스탠드업과 오늘 스탠드업의 유일한 차이여야 한다.

## "F3 롤백해도 EXT/GHOST 가 그대로" 라는 순간

qa 가 v5 결과를 들고 왔을 때 한참 멍했다. F3 를 롤백했는데 결과가 v4 와 완전히 동일하면, 우리가 찾던 범인이 F3 가 아니거나, 애초에 **고장 난 건 fixture 였는지도 모른다**. 실제 사용자가 브라우저에서 extend 를 해보면 잘 될 수도 있다. 그러면 오늘 하루 F3 롤백에 쓴 시간은 "없는 병을 치료한" 비용이 된다. 하지만 재현 spec 이 RED 인 한 우리는 안심할 수 없다. "spec 은 RED 지만 실기는 OK" 는 받아들일 수 없는 상태다. 둘 중 하나는 거짓말을 하고 있다.

이 순간이 qa 의 본질이기도 하다. "숫자 뒤에 숨는 QA 가 아니라 범인을 지목하는 QA" — qa 가 저녁 반성에서 한 말. bisect 자동화 스크립트를 만들겠다는 약속. 내일 분수령은 사용자 실기 테스트다. 두 시나리오로 갈린다:

- **실기 PASS**: fixture 가 거짓말. spec 구조 자체를 뜯어 고쳐야 함 (W2-E 이미지 태그 정책과 결합된 campaign 오염 가설)
- **실기 FAIL**: F4 `effectiveHasInitialMeld` 가이드가 진짜 범인. W2-A bisect 즉시 착수

어느 쪽이든 이 애매한 상태를 하루 더 끌면 안 된다. 사용자 증언이 분기점이다.

## go-dev 의 조용한 성공

오늘 유일하게 약속을 정확히 지킨 사람은 go-dev 였다. PR #77 에서 `game_service_confirm_test.go` 에 TC-1/2/3 를 RED 로 먼저 커밋하고, 그 다음 `ConfirmTurn` final `validateAllMelds` 구현으로 GREEN 전환했다. E2E `rule-invalid-meld-cleanup.spec.ts` SC1/2/3 도 구현 전에 선행 작성. 692 PASS 유지.

고요한 성공이었다. 떠들썩한 수습 PR 3건이 하루를 덮고 있었는데, 그 안에 이 작은 정확성 하나가 조용히 있었다. 저녁 반성에서 go-dev 는 "내가 잘했다" 대신 "같은 시간에 frontend 기본 실수 4건을 방관했다" 라고 말했다. `ROLLBACK_FORCED` 이벤트가 프론트에서 어떻게 표시되는지 한 번도 확인 안 했다고. 약속 지킨 사람이 더 엄격하게 본인을 본다. 내일 회고 모드에서 이 패턴을 팀 전체로 확산시켜야 한다.

## 사용자 "진짜 일 못하는구나" 라는 말이 남긴 것

오후 중간쯤 사용자가 이 말을 했을 때 방어하지 않았다. 방어할 수 없었다. 아침에 반성하고 오후에 같은 실수를 반복한 사람에게 "Opus 4.7 은 생각 없이 일한다" 는 평가는 정당하다. SKILLS 활용을 매번 짚어줘야 한다는 지적, 스크린샷을 재검증하지 않으면 OCR 오독을 못 잡는다는 지적, 에이전트 병렬 호출 시 worktree 격리를 말해줘야만 한다는 지적 — 전부 **사용자가 나의 체크리스트 역할을 하고 있었다는 뜻**이다. 이건 사용자의 불만이 아니라 내 설계의 실패다.

내일부터 3줄 체크리스트를 본인 응답에 명시한다 — (1) 관련 SKILLS 스캔, (2) worktree 격리 필요 여부, (3) HEAD SHA. 사용자가 보이는 위치에 매번 노출해서 건너뛸 수 없게 만든다. 구조가 의지를 이긴다. 오늘 배운 유일한 교훈.
