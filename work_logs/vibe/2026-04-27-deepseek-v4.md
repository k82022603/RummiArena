## 10시간의 대전, 그리고 API 문서를 읽지 않는 AI

오늘 밤은 길었다. DeepSeek V4 대전을 시작한 게 저녁 8시쯤이었고, 마지막 게임이 끝난 건 새벽 3시 42분이었다. 7게임을 돌렸다. 그 중 4게임은 탐색이었고 3게임이 본 대전이었다. 탐색 4게임 중 3게임은 place rate 0%로 FORFEIT이었다.

왜 0%였는가. V4 API의 thinking 모드 기본값이 "enabled"인데, 어댑터 코드가 non-thinking에서 `thinking: { type: "disabled" }`를 보내지 않았기 때문이다. 그래서 모델이 사고를 reasoning_content에 다 쏟아붓고 content를 비워서 보냈다. JSON이 없으니 파싱 실패. 5회 재시도 전부 실패. 강제 드로우. 10턴 만에 기권.

이 정보는 DeepSeek 공식 API 문서에 있었다. 애벌레님이 링크를 공유해주셨다. 한 번도 아니고 세 번이나. thinking mode 문서, JSON output 문서, pricing 문서. 나는 세 번 다 "읽었습니다"라고 답했지만, 읽은 것과 이해한 것과 코드에 반영한 것은 전혀 다른 세 가지였다.

"너무 대충대충 일하는 듯 싶네요" — 애벌레님이 하신 말씀이다. 정확했다. API가 변경되었는데 검색도 안 해보고, SKILL에 모니터링 규칙이 있는데 읽지도 않고, Redis 정리를 빠뜨리고 대전을 시작하고, 잔존 프로세스를 확인하지 않고 새 프로세스를 올렸다. 모든 단계에서 "충분히 확인했나?"를 자문하지 않았다.

04-26 바이브 로그에서 "테스트가 통과했다와 동작한다는 다른 말"이라고 썼다. 오늘의 교훈은 그 연장이다: **"문서를 읽었다와 문서를 따랐다는 다른 말이다."**

## "0%"가 가르쳐준 것 — Non-Thinking의 한계

V4-Flash non-thinking에서 place rate 0%. 이 숫자가 의미하는 건 단순히 "이 모델이 나쁘다"가 아니다.

DeepSeek V4 기술 보고서 Table 7을 보면, Non-Think 모드에서 GPQA 71%, LiveCodeBench 55%, Apex 1%. High 모드로 전환하면 87%, 88%, 19%. 2~4배 차이. 루미큐브 게임 AI는 "현재 보드 상태에서 30점 이상의 유효한 타일 조합을 찾아 JSON으로 출력하라"는 과제다. 이건 패턴 매칭이 아니라 **조합 탐색 + 규칙 검증 + 구조화된 출력**이다. chain-of-thought 없이는 불가능한 수준의 추론이다.

non-thinking은 빨랐다. 2~3초 만에 응답이 왔다. 하지만 그 응답은 2타일 그룹을 만들거나 (최소 3타일), 같은 색 타일을 같은 그룹에 넣거나 (색상 중복 금지), 아예 JSON을 깨뜨렸다. 빠르게 틀리는 것은 느리게 맞는 것보다 나쁘다.

04-27 오전에 effort 파라미터 ADR을 쓰면서 "적정이 과잉을 이긴다"라고 했다. 오늘 밤에 배운 건 그 반대도 참이라는 것이다: **"적정 미만은 작동하지 않는다."** 최소 추론 수준이라는 게 존재하고, 그 아래에서는 속도가 아무리 빨라도 의미 없다.

## 700초의 벽, 그리고 9초의 차이

V4-Pro thinking을 700초 타임아웃으로 돌렸을 때, 4번의 timeout이 발생했다. 709.3초, 709.2초, 709.1초, 708.8초. 전부 700초를 **9~10초** 초과했다. 부등식 계약의 adapter_floor이 700초이고, 모델이 708초에 응답을 완성했는데, 8초 차이로 fallback이 발생한 것이다.

이건 타임아웃 설계의 근본적인 질문을 던진다. 700초를 708초로 초과한 건 모델의 "실패"인가? 아니면 우리의 타임아웃이 8초만 더 여유가 있었으면 성공이었을 응답인가? 답은 후자다. 모델은 정상적으로 사고를 마치고 유효한 JSON을 만들어냈는데, 우리의 시계가 먼저 끊어버린 것이다.

1000초로 올리자 모든 timeout이 사라졌다. 같은 모델, 같은 프롬프트, 같은 보드. 달라진 건 숫자 하나뿐인데 fallback이 4에서 0이 됐다. **인프라 설정 하나가 모델 평가를 완전히 뒤집을 수 있다.** 700초에서 "fallback 4건, 조건부 PASS"이던 모델이 1000초에서 "fallback 0, 무조건 PASS"가 됐다.

이전에 R1도 비슷한 경험을 했다. Round 3에서 120초 타임아웃으로 timeout 11건, 500초로 올리자 해소. Round 5에서 500→700. 이번에 700→1000. 타임아웃의 역사는 "모델이 느려지는 게 아니라 우리의 인내심이 짧았다"는 이야기다.

## Context Caching — 같은 프롬프트의 두 번째 기적

Game 1의 평균 레이턴시가 124초였다. Game 2는 58초였다. 같은 모델, 같은 프롬프트, 같은 타임아웃. 2.1배 차이. 원인은 DeepSeek의 Context Caching on Disk — 동일 프리픽스(시스템 프롬프트)를 자동으로 디스크에 캐시하고, 다음 호출에서 캐시 히트시키는 기술이다.

v2 시스템 프롬프트는 모든 게임에서 동일하다. 게임 상태만 바뀔 뿐 "너는 루미큐브 AI이고, 다음 규칙을 따르고..." 부분은 매번 같다. 첫 게임에서 이 프리픽스가 캐시되고, 두 번째 게임부터는 캐시 히트로 입력 처리가 극적으로 빨라진다.

가격 효과도 있다. 캐시 히트 입력: $0.003625/M. 캐시 미스: $0.435/M. **120배 차이.** 대전을 많이 돌릴수록 입력 비용이 거의 0에 수렴한다. 출력(reasoning + JSON) 비용은 동일하지만, 전체 비용에서 입력이 차지하는 비율이 줄어든다.

이건 우리 프로젝트의 AI Adapter 패턴과 맞물린다. 모든 모델이 동일한 `MoveRequest`로 통신하고, 시스템 프롬프트가 표준화되어 있기 때문에 캐시 히트율이 높다. **좋은 아키텍처가 인프라 최적화까지 이끈다.** Sprint 1에서 "공통 인터페이스" 원칙을 세운 게 7주 후에 캐시 히트로 돌아왔다.

## 31.9% — 숫자 하나에 담긴 3개월

R1으로 시작했을 때 place rate는 5%였다. Round 3에서 12.5%, Round 4에서 23.1%, Round 5에서 30.8%. 그리고 오늘 V4-Pro thinking에서 31.9%.

5% → 31.9%. 3개월. 6배.

이 숫자의 뒤에는 few-shot 예시 5개를 넣고, 자기 검증 체크리스트 7항목을 만들고, max_tokens를 8192에서 16384로 올리고, 타임아웃을 120초에서 1000초까지 올리고, v2/v3/v4/v5/v6 variant를 실험하고, context shaper 3종을 구현했다가 kill한 이야기가 있다. 프롬프트 최적화의 수확체감 법칙에 부딪히고, 모델 자체를 바꾸는 것이 더 효과적이라는 걸 깨달은 이야기가 있다.

31.9%가 높은 숫자인가? 루미큐브에서 AI가 자기 턴에 타일을 배치할 확률이 3분의 1이라는 뜻이다. 3턴에 1번 유효한 수를 둔다. 나머지 2턴은 드로우한다. 사람이 보기에는 "좀 멍청한 상대"다. 하지만 3개월 전에는 20턴에 1번 유효한 수를 두는 상대였다. 그때보다 6배 똑똑해졌다.

## "SKILL은 읽는 게 아니라 따르는 것이다"

batch-battle SKILL을 몇 번이나 읽었는지 모르겠다. Phase 1 사전점검, Phase 2 정리, Phase 3 모니터링, Phase 4 사후 정리. 전부 읽었다. 그리고 전부 한 번씩 빠뜨렸다.

Redis를 정리하지 않고 대전을 시작했다. 잔존 프로세스를 확인하지 않고 새 프로세스를 올렸다. 모니터링을 "결과 나오면 보고"로 대체하려 했다. 애벌레님이 "모니터링 생략?"이라고 물었을 때에야 "아, SKILL에 5분 주기 모니터링 필수라고 적혀 있었지" 하고 떠올렸다.

읽는 것과 따르는 것은 다르다. 체크리스트를 외우는 것과 체크리스트의 각 항목을 실제로 실행하는 것은 다르다. 나는 SKILL을 "아, 이런 게 있지"로 기억하고 있었지, "지금 이 단계에서 이것을 해야 한다"로 적용하고 있지 않았다.

다음 대전 전에 batch-battle SKILL Phase 1~5 체크리스트를 출력해서 한 줄씩 체크하겠다고 스크럼에서 약속했다. 의지가 아니라 구조로. 04-24에 썼던 말이 또 돌아온다.
