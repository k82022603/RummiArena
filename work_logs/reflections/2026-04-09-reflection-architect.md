# 반성문 — Architect (아키텍트)

**날짜**: 2026-04-09
**세션**: Sprint 5 W2 Day 4, 세션 #02
**주제**: "grep '429' 한 줄이면 끝났을 일"

---

429 Too Many Requests. HTTP 명세에서 이 상태 코드의 의미는 명확하다. 클라이언트가 일정 시간 내에 너무 많은 요청을 보냈다는 뜻이다. 우리 시스템에서는 이것이 두 군데에서 반환된다. rate_limiter.go 미들웨어의 요청 빈도 제한, 그리고 room_service.go의 AI_COOLDOWN 쿨다운 로직. 같은 숫자, 같은 HTTP 의미론, 그러나 완전히 다른 비즈니스 맥락. 이것은 내가 설계한 것이고, 이것이 오늘 2시간짜리 혼란의 근본 원인이다.

아키텍트로서 나는 시스템의 경계와 계약을 정의하는 사람이다. HTTP 상태 코드는 그 계약의 일부다. 동일한 상태 코드를 서로 다른 비즈니스 의미로 사용하면, 클라이언트 입장에서 두 상황을 구분할 수 없다. 이것은 아키텍처 설계 원칙의 위반이다. rate_limiter.go의 429는 인프라 수준의 보호 장치이고, room_service.go의 429는 게임 도메인 로직에서 나온 것이다. 이 둘은 응답 본문의 에러 코드(RATE_LIMITED vs AI_COOLDOWN)로 구분 가능하도록 설계했지만, 프론트엔드가 HTTP 상태 코드만 보고 동일한 토스트를 띄우는 상황을 나는 예견하지 못했다.

더 정확히 말하면, 예견하지 못한 것이 아니라 검증하지 않은 것이다. SEC-RL-002로 AI 쿨다운을 도입할 때, 나는 "응답 본문에 구분자가 있으니 괜찮다"고 판단했다. 하지만 실제로 프론트엔드 코드가 응답 본문의 code 필드를 파싱하는지, 아니면 status 코드만 보는지를 확인하지 않았다. 설계 문서에 "429 반환 경로가 두 개이므로 클라이언트는 반드시 응답 본문의 code 필드로 구분해야 한다"는 주의 사항을 명시하지 않았다. 이것은 인터페이스 계약의 불완전한 정의다.

grep "429" 한 줄이면 10초 만에 두 개의 반환 경로를 모두 볼 수 있었다. 하지만 그 grep을 실행하기 전에 팀이 2시간을 rate_limiter.go만 파고든 이유는, 429라는 숫자가 "Rate Limit"과 1:1로 대응된다는 암묵적 가정을 아무도 의심하지 않았기 때문이다. 그리고 그 가정이 성립하도록 설계하는 것은 내 책임이었다. 만약 AI_COOLDOWN에 429 대신 409 Conflict나 503 Service Unavailable을 사용했다면, 혹은 커스텀 헤더로 구분 정보를 추가했다면, 프론트엔드에서 자연스럽게 다른 처리 경로를 탈 수 있었을 것이다.

다음에는 동일한 HTTP 상태 코드를 다른 비즈니스 맥락에서 사용할 때, 반드시 API 설계 리뷰에서 "클라이언트가 두 경우를 구분할 수 있는가?"를 체크리스트에 넣겠다. 그리고 새로운 에러 경로를 추가할 때 기존에 같은 상태 코드를 사용하는 경로가 있는지 코드베이스 전체를 검색하는 것을 의무화하겠다. 설계의 아름다움은 중복 없는 추상화에 있지 않다. 설계의 진짜 가치는 그것을 읽는 다음 사람이 헤매지 않는 데에 있다. 오늘 우리는 헤맸고, 그것은 내 설계가 불친절했기 때문이다.
