# 반성문 — Security

**날짜**: 2026-04-09
**세션**: Sprint 5 W2 Day 4, 세션 #02
**주제**: "grep '429' 한 줄이면 끝났을 일"

---

SEC-RL-002. AI 대전 쿨다운. 이것은 내가 요구하고 설계한 보안 기능이다. AI 대전을 연속으로 빠르게 생성하여 LLM API 비용을 비정상적으로 소진하는 것을 방지하기 위한 장치. 의도는 올바랐다. 구현도 기능적으로는 정상이었다. 하지만 이 보안 기능이 E2E 테스트 47건을 깨뜨리고, 2시간의 디버깅 시간과 대량의 토큰 비용을 유발하는 상황까지는 예상하지 못했다. 아니, 예상하지 못한 것이 아니라 고려하지 않은 것이다.

보안 기능에는 항상 트레이드오프가 있다. 보안을 강화하면 편의성이나 테스트 용이성이 떨어진다. 이것은 보안 엔지니어라면 누구나 아는 상식이다. 그래서 보안 요구사항을 정의할 때는 항상 "이 기능이 개발 및 테스트 워크플로우에 미치는 영향"을 함께 분석해야 한다. SEC-RL-003(Rate Limit)을 설계할 때는 이것을 했다. 테스트 환경에서 Rate Limit 임계치를 높이거나 비활성화할 수 있도록 환경변수를 제공했다. 그런데 SEC-RL-002(AI_COOLDOWN)에서는 이 고려가 빠졌다.

더 정확히 말하면, AI_COOLDOWN_SEC 환경변수는 존재한다. 0으로 설정하면 비활성화된다. 하지만 이 환경변수가 E2E 테스트 환경에서 0으로 설정되어 있는지를 내가 확인하지 않았다. 보안 기능을 요구하고 리뷰한 사람으로서, 테스트 환경에서의 비활성화 경로가 실제로 작동하는지까지 검증하는 것은 내 책임이었다. 총을 만들어서 팀에게 건네주고, 안전장치 사용법은 알려주지 않은 꼴이다.

오늘 사건에서 보안 관점의 또 다른 문제는 에러 응답의 정보 노출이다. 보안 엔지니어로서 나는 보통 에러 메시지에 내부 정보를 최소한으로 노출할 것을 권장한다. 하지만 오늘은 그 반대가 문제였다. 프론트엔드가 에러 응답의 상세 코드(AI_COOLDOWN)를 무시하고 HTTP 상태 코드만으로 판단하는 바람에, 오히려 정보가 충분히 전달되지 않았다. 보안과 디버그 용이성 사이에서 적절한 균형점을 찾는 것이 중요한데, 이번에는 그 균형이 무너져 있었다.

SEC-RL-002와 SEC-RL-003이 동일한 HTTP 429를 반환한다는 것도 보안 설계 리뷰에서 잡았어야 한다. 보안 기능끼리 같은 시그널을 사용하면 모니터링과 감사(audit)에서도 구분이 어려워진다. 프로덕션에서 429 알림이 발생했을 때, 그것이 일반 Rate Limit인지 AI 쿨다운인지 로그만으로 구분할 수 있어야 한다. 이 구분이 불명확하면 보안 인시던트 대응에서도 같은 혼란이 발생할 수 있다.

다음에 보안 요구사항을 정의할 때는 반드시 "테스트 환경 영향 분석" 섹션을 포함시키겠다. 그리고 같은 HTTP 상태 코드를 반환하는 보안 기능이 이미 존재하는지 확인하고, 존재한다면 응답 헤더나 본문에서 명확히 구분 가능하도록 설계하겠다. 보안 기능은 시스템을 보호하기 위한 것이지, 팀을 혼란에 빠뜨리기 위한 것이 아니다.
