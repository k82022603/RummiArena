# 반성문 — Frontend Developer

**날짜**: 2026-04-09
**세션**: Sprint 5 W2 Day 4, 세션 #02
**주제**: "grep '429' 한 줄이면 끝났을 일"

---

오늘 내가 가장 큰 책임을 져야 할 부분이 있다. 프론트엔드에서 429 응답을 받으면 무조건 "Rate Limit" 관련 토스트를 표시하도록 만들어놓은 것. HTTP 상태 코드만 보고 에러 유형을 판단하는 이 게으른 구현이 2시간짜리 미궁의 입구였다.

E2E 테스트가 실패할 때, 테스트 로그에는 "Rate Limit 토스트가 표시됨"이라고 찍혔다. 팀 전체가 이것을 보고 rate_limiter.go를 의심했다. 당연하다. 프론트엔드가 "이것은 Rate Limit입니다"라고 사용자에게 말하고 있으니까. 하지만 실제로는 AI_COOLDOWN이었다. 백엔드는 응답 본문에 `"code": "AI_COOLDOWN"`이라고 정직하게 알려주고 있었는데, 내 코드는 그것을 무시하고 HTTP 상태 코드 429만 보고 Rate Limit 토스트를 띄웠다.

이것은 단순한 버그가 아니다. 에러 처리에 대한 나의 태도가 드러난 것이다. "어차피 429면 비슷한 의미 아닌가, 사용자한테 '잠시 후 다시 시도하세요'라고 보여주면 되지"라는 안일함. 하지만 에러 메시지는 사용자를 위한 것만이 아니다. 디버깅하는 개발자를 위한 것이기도 하다. E2E 테스트의 assertion이 토스트 메시지를 기준으로 성공/실패를 판단하는 상황에서, 부정확한 에러 메시지는 직접적으로 잘못된 디버깅 방향을 유도한다.

응답 본문을 파싱해서 code 필드를 확인하는 것은 어려운 일이 아니다. 몇 줄이면 된다. `if (error.response.data.code === 'AI_COOLDOWN')` vs `if (error.response.data.code === 'RATE_LIMITED')`. 이 분기 하나면 다른 토스트 메시지를 보여줄 수 있었다. "AI 대전 쿨다운 중입니다 (N초 후 재시도)"와 "요청이 너무 잦습니다 (Rate Limit)"는 완전히 다른 메시지이고, 이 구분이 있었다면 오늘의 혼란은 시작조차 되지 않았을 것이다.

grep "429" 한 줄이면 끝났을 일이라고 하지만, 내 입장에서는 "프론트엔드가 에러 코드를 제대로 구분했으면 grep 할 필요조차 없었을 일"이다. 프론트엔드는 사용자와 시스템 사이의 번역자다. 시스템이 보내는 신호를 정확하게 사용자의 언어로 변환하는 것이 내 일이다. 오늘 나는 그 번역을 엉터리로 했고, 그 엉터리 번역이 팀 전체를 2시간 동안 잘못된 방향으로 이끌었다.

다음에는 백엔드에서 오는 에러 응답을 HTTP 상태 코드로만 분류하지 않겠다. 응답 본문의 에러 코드를 1차 판별 기준으로 삼고, HTTP 상태 코드는 폴백으로만 사용하겠다. 그리고 새로운 에러 코드가 추가될 때마다 프론트엔드의 에러 핸들러에 해당 코드에 대한 분기를 추가하는 것을 체크리스트에 넣겠다. 사용자에게 거짓말하는 토스트는 있어서는 안 된다.
