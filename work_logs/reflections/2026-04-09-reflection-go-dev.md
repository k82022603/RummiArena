# 반성문 — Go Developer

**날짜**: 2026-04-09
**세션**: Sprint 5 W2 Day 4, 세션 #02
**주제**: "grep '429' 한 줄이면 끝났을 일"

---

room_service.go 97번째 줄. Status: 429. 이것을 작성한 사람이 나다. AI_COOLDOWN 로직을 SEC-RL-002 보안 요구사항에 따라 구현하면서, HTTP 429를 반환하도록 만든 것도 나다. 그리고 오늘 E2E 47건 실패 원인을 디버깅하면서 2시간 동안 rate_limiter.go만 들여다보고 있었던 것도 나다. 내가 직접 작성한 코드에서 429를 반환하고 있다는 사실을 잊고 있었다.

이 고백이 가장 부끄러운 부분이다. rate_limiter.go는 미들웨어다. 요청이 핸들러에 도달하기 전에 동작한다. room_service.go의 AI_COOLDOWN은 서비스 레이어에서 동작한다. 미들웨어를 통과한 뒤에 발생하는 429다. 두 경로는 완전히 다른 계층에 있고, 동작 조건도 다르다. Rate Limiter는 단위 시간당 요청 수를 세고, AI_COOLDOWN은 마지막 AI 대전 생성 시각으로부터의 경과 시간을 본다. E2E 테스트에서 AI 대전 방을 빠르게 연속 생성하면 Rate Limiter보다 AI_COOLDOWN에 먼저 걸릴 수 있다. 이것을 나는 코드를 작성할 때 알고 있었다.

그런데 왜 디버깅할 때는 몰랐는가. 나는 "429 = Rate Limit"이라는 단축 회로가 머릿속에 형성되어 있었다. rate_limiter.go를 Sprint 5에서 대대적으로 작업했기 때문에, 429라는 숫자를 보면 자동으로 그 파일이 떠올랐다. AI_COOLDOWN은 Sprint 4 말에 추가한 것이라 상대적으로 기억에서 멀어져 있었다. 하지만 이것은 변명이다. 프로페셔널한 개발자라면 자기가 작성한 코드베이스에서 특정 값이 어디서 사용되는지를 기억에 의존하지 않는다. 도구를 쓴다. grep을 쓴다.

`grep -rn "429" src/game-server/` 한 줄이면 rate_limiter.go와 room_service.go가 나란히 나왔을 것이다. 아, 여기에도 429가 있었지. 그 한마디면 끝났다. 나는 그 한 줄을 치지 않았다. 대신 rate_limiter.go의 윈도우 크기를 조정하고, 환경변수를 바꾸고, E2E 테스트를 다시 돌리고, 실패하고, 또 rate_limiter.go를 열고. 이 루프를 몇 바퀴 돌았는지 모르겠다.

쿨다운 로직을 room_service.go에 넣은 것 자체가 문제였다고는 생각하지 않는다. 비즈니스 규칙이니까 서비스 레이어에 있는 것이 맞다. 하지만 같은 HTTP 상태 코드를 사용한 것은 의심의 여지 없이 내 판단 착오다. "쿨다운이니까 too many requests니까 429지"라고 생각했다. 의미론적으로 틀리지는 않지만, 디버깅 편의성과 클라이언트 구분 가능성을 전혀 고려하지 않았다.

다음에는 새로운 에러 응답을 추가하기 전에 반드시 기존 코드베이스에서 같은 상태 코드를 사용하는 곳을 검색하겠다. 그리고 디버깅 시 "이 값이 반환되는 모든 경로"를 열거하는 것을 첫 번째 단계로 삼겠다. 기억에 의존하지 않겠다. 기억은 배신하지만 grep은 배신하지 않는다.
