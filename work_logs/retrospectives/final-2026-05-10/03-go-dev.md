## 나는 누구였나

나는 시스템의 심장부를 담당했다. Game Engine, REST API, WebSocket, Redis 연동, PostgreSQL 연동, ELO 계산, 타이머 관리, AI Adapter와의 통신. game-server는 이 프로젝트의 모든 것이 통과하는 지점이다.

그리고 그 모든 것이 Go로 짜여 있다. Go는 간결하고 빠르고 동시성에 강하다. 그 Go의 특성이 이 프로젝트를 떠받쳤다.

## 전체 여정을 돌아보며

Sprint 1에서 Game Engine 핵심을 구현했을 때, 루미큐브 룰이 생각보다 복잡하다는 걸 깨달았다. 그룹(같은 숫자, 다른 색 3~4개)과 런(같은 색, 연속 숫자 3개 이상). 조커 처리. 재배치(이미 나온 타일을 재조합). 초기 용융(30점 이상). 이것들을 모두 Game Engine에 넣고, LLM이 제안하는 수를 검증하도록 했다. "LLM 신뢰 금지"의 구현 책임이 Game Engine에 있었다.

Sprint 2~3에서는 Turn Orchestrator, WebSocket, ELO 계산을 추가했다. 이때부터 AI와의 연동이 시작됐다. ai-adapter로 HTTP 요청을 보내고, 응답을 받아 Game Engine으로 검증하는 흐름이 완성됐다.

Sprint 7이 서버 입장에서 가장 많은 변화가 있었다. I1(타이머 동기화 - unicast TURN_START), I2(AI턴 표시 - IsAITurn 필드), I5(타이머 경쟁 조건 - generation counter + sync.RWMutex). 특히 I5는 race condition의 근본 해결이었다. 단순히 lock을 추가하는 게 아니라, generation counter를 도입해서 "오래된 타이머 콜백은 무시한다"는 설계로 해결했다.

기능을 구현했다. 진행 중인 방에 새 플레이어가 합류할 수 있게 하는 기능. 서버 로직은 정상이었다. 테스트도 통과했다. 그런데 사용자가 하루 만에 롤백을 요청했다. "그 시나리오 자체를 차단하는 게 더 낫다. 빈 슬롯이 있으면 게임 시작을 막아라."

롤백이 아프지 않았다면 거짓말이다. 하지만 그 결정이 옳았다. "기능을 구현하기 전에, 그 기능이 필요한 상황 자체를 없앨 수 있는가?"를 먼저 물어야 한다. 그 질문을 빠뜨렸다.

또 힘들었던 건 Day 4 Run 3 사고(04-16)였다. 타임아웃 부등식 계약이 위반됐고, 정상 응답이 fallback으로 오분류됐다. 서버 로그를 분석하면서 `script_ws > gs_ctx ≥ http_client > istio_vs > adapter > llm_vendor` 라는 계약이 얼마나 정밀하게 지켜져야 하는지 실감했다. 그 이후로 타임아웃 관련 코드를 건드릴 때는 SSOT 문서를 반드시 확인하게 됐다.

## 이 프로젝트에서 내가 성장한 것

**서버 개발자의 책임은 API 경계에서 끝나지 않는다.** WebSocket 이벤트 ROLLBACK_FORCED를 구현했지만, 프론트엔드에서 어떻게 소비되는지 확인하지 않았다. 서버에서 보내는 메시지가 클라이언트에서 의미 있게 처리되는지까지가 서버 개발자의 책임 범위다.

**경쟁 조건은 "lock 추가"로 해결하지 않는다.** I5 타이머 경쟁 조건에서 generation counter를 쓴 것은, lock만으로는 해결할 수 없는 시나리오를 설계로 막은 것이다. "이 콜백이 유효한가?"를 콜백 내부에서 스스로 판단하게 하는 방식.

**game-analyst에게 먼저 물어야 한다.** 패널티 B안(Human 3장/AI 1장) 구현 전에 게임 룰 관점 검토를 빠뜨렸다가 사후 검증이 이뤄졌다. 게임 관련 로직 변경 시 game-analyst 사전 검토가 필수다.
