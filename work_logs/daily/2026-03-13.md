## 발생 이슈 (6건)

### ISS-01: Helm chart 미작성 (Sprint 0 누락)

| 항목 | 내용 |
|------|------|
| 심각도 | Medium |
| 상태 | 해결 완료 |
| 발생 시점 | K8s 전환 착수 시 |
| 영향 | Helm chart templates 없이 배포 불가 |

**상세**: `helm/charts/` 디렉토리에 스캐폴딩(Chart.yaml, values.yaml)만 있었고, 실제 templates(deployment, service, configmap 등)가 없었다. Sprint 0에서 Helm chart 작성이 누락된 것으로 확인됨.

**대응**: 5개 서비스 전체에 대해 Helm chart templates를 신규 작성하여 해결.

**회고**: Sprint 0 WBS에서 "Helm chart 작성"이 스캐폴딩 수준으로만 정의되어 있었다. 향후 인프라 태스크는 "동작 확인"까지를 완료 기준(DoD)에 포함해야 한다.

### ISS-02: game-server init container ResourceQuota 위반

| 항목 | 내용 |
|------|------|
| 심각도 | High |
| 상태 | 해결 완료 |
| 발생 시점 | game-server Pod 생성 시 |
| 영향 | Pod 생성 실패 (CreateContainerConfigError) |

**상세**: game-server의 init container(wait-for-postgres, wait-for-redis)에 resource limits가 설정되지 않아 Pod 생성이 실패했다. rummikub namespace의 ResourceQuota(`rummikub-quota`)가 모든 컨테이너에 limits 설정을 필수로 요구하고 있었기 때문이다.

**대응**: init container에 `resources.requests`(cpu: 50m, memory: 32Mi) 및 `resources.limits`(cpu: 100m, memory: 64Mi)를 명시하여 해결.

**교훈**: K8s ResourceQuota가 활성화된 namespace에서는 init container에도 반드시 resource limits를 설정해야 한다. 향후 Helm chart 템플릿에 init container resource 설정을 기본값으로 포함할 것.

### ISS-03: Frontend NEXTAUTH_SECRET 미설정

| 항목 | 내용 |
|------|------|
| 심각도 | High |
| 상태 | 해결 완료 |
| 발생 시점 | Frontend Pod 기동 시 |
| 영향 | NextAuth.js 500 에러 발생 |

**상세**: K8s 환경에서 Frontend Pod에 `NEXTAUTH_SECRET` 환경변수가 누락되어 NextAuth.js가 500 에러를 반환했다. docker-compose에서는 `.env.local`로 주입되었지만 K8s ConfigMap에는 포함되지 않았다.

**대응**: frontend ConfigMap에 `NEXTAUTH_SECRET` 키를 추가하여 해결.

**교훈**: 로컬 개발 환경(.env.local)과 K8s 환경(ConfigMap/Secret)의 환경변수를 동기화하는 체크리스트가 필요하다. 시크릿성 값은 K8s Secret으로 분리하는 것이 바람직하다.

### ISS-04: ConfigMap 템플릿 하드코딩 문제

| 항목 | 내용 |
|------|------|
| 심각도 | Low |
| 상태 | 해결 완료 |
| 발생 시점 | Frontend ConfigMap 업데이트 시 |
| 영향 | 새 환경변수 추가 시마다 템플릿 수정 필요 |

**상세**: frontend `configmap.yaml` 템플릿이 3개 키만 하드코딩하고 있어, 새 환경변수를 추가할 때마다 템플릿 파일을 직접 수정해야 했다.

**대응**: Helm 템플릿의 `range` 루프(`{{- range $key, $value := .Values.env }}`)로 변경하여 `values.yaml`의 `env` 맵에서 동적으로 키를 생성하도록 개선.

### ISS-05: DB 마이그레이션 down.sql 실행 이슈

| 항목 | 내용 |
|------|------|
| 심각도 | High |
| 상태 | 해결 완료 |
| 발생 시점 | PostgreSQL 마이그레이션 실행 시 |
| 영향 | 마이그레이션 후 테이블이 존재하지 않음 |

**상세**: migration 파일 glob 패턴(`*.sql`)이 `down.sql` 파일을 먼저 실행하여 테이블을 DROP한 뒤 `up.sql`로 CREATE하는 순서가 보장되지 않았다. 결과적으로 up.sql 실행 후 down.sql이 실행되어 테이블이 삭제된 상태로 마이그레이션이 종료되었다.

**대응**: `up.sql` 파일만 별도로 지정하여 실행하는 방식으로 변경하여 해결.

**교훈**: 마이그레이션 도구(golang-migrate 등)를 도입하면 up/down 순서가 자동 관리된다. Sprint 2에서 정식 마이그레이션 도구 도입을 검토한다.

### ISS-06: DevOps 에이전트 Bash 권한 부재

| 항목 | 내용 |
|------|------|
| 심각도 | Medium |
| 상태 | 우회 해결 |
| 발생 시점 | K8s 전환 세션 전반 |
| 영향 | helm install/kubectl 명령 직접 실행 불가 |

**상세**: Agent에 Bash 권한이 없어 `helm install`, `kubectl apply` 등의 명령을 직접 실행할 수 없었다. 파일 생성(Helm chart, YAML)만 가능한 상태였다.

**대응**: 에이전트가 Helm chart 파일을 생성하고, 사용자(애벌레)가 직접 명령을 실행하는 분업 방식으로 우회. 향후 DevOps 에이전트에 제한적 Bash 권한(helm, kubectl만)을 부여하는 방안을 검토한다.
