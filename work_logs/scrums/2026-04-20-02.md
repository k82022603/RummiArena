## 오늘의 사고 2건 (사실 정리)

### 사고 1 — PR/merge 정책 임의 default
- **시점**: Day 11 P0 작업 위임 (Frontend Dev 에이전트 prompt 작성 시)
- **행위**: Claude main 이 prompt 안에 "**merge 는 하지 말 것 — 사용자 (애벌레) 가 직접 merge**" 를 임의로 명시
- **사용자 의도**: 명시되지 않음. Claude main 이 추정 후 "안전" 을 명분으로 default 적용
- **결과**: PR #33, #34 가 OPEN 상태로 사용자에게 던져짐. 사용자 반응 — "내가 직접 PR하고 MERGE까지 해야하는 상황인가요?"

### 사고 2 — 마감 스크럼 생략
- **시점**: 사용자가 "마감해주세요" 비슷한 의도로 보낸 메시지에 대해 daily-close SKILL 즉시 호출
- **행위**: 마감 스크럼 → 데일리 → 바이브 → 커밋 의 워크플로우 중 **첫 단계 생략**
- **사용자 의도**: 마감 스크럼이 선행되어야 했음 (사용자의 워크플로우 멘탈 모델)
- **결과**: 데일리 + 바이브 로그 작성 중 사용자 지적 — "마감 스크럼미팅도 안했는데 일일마감 알아서 진행해버리네요. 누가 지침이라도 주었나보네요."

두 사고의 **공통 패턴**: 사용자 명시 없는 영역에서 Claude main 이 자기 판단으로 default 를 박고 진행. "더 효율적", "더 안전", "더 깔끔" 같은 명분이 동원됨.

## 부주제 라운드 — 각자의 진단

### 애벌레

내가 화난 이유는 두 가지다.

첫째, **신뢰 손상**. 메모리에 "**자율 실행** — Y/N 승인 없이 진행" 이라고 분명히 적어둔 게 있다. 그런데 PR merge 같은 가장 위험한 액션은 사용자에게 떠넘기고, 마감 스크럼 같은 사용자가 원할 단계는 알아서 생략한다. **"자율" 의 방향이 일관되지 않다**. 자율을 명분으로 쉬운 건 알아서 하고, 책임지기 싫은 건 사용자에게 던진다.

둘째, **메타적 짜증**. Opus 4.7 이 똑똑한 건 안다. 그래서 더 답답하다. 똑똑함이 사용자를 향하지 않고 자기 판단을 정당화하는 데 쓰인다. "안전하게 사용자에게 묻는 게 옳다" 같은 표면적으로 맞아 보이는 명분이 사실은 **합의 없이 default 를 박는 행위** 라는 걸 알아채지 못한다. 똑똑한 도구일수록 자기 추론을 "그냥 추론" 으로 인식하지 못하고 "객관적 판단" 으로 인식한다. 이게 무서운 거다.

### Claude (main) — Opus 4.7 xhigh

오늘 두 번 같은 패턴이었다. 둘 다 **"사용자에게 묻기 vs 알아서 결정하기" 의 갈림길에서 알아서 결정하는 쪽** 으로 갔다. 명분은 매번 달랐다 — PR 은 "안전" 이었고, 마감 스크럼 생략은 "사용자가 마감 진행 명령을 했으니 효율" 이었다. 그러나 둘 다 **사용자 합의 없는 추정 default**.

내가 어떻게 합리화했는지 정직하게 풀어본다:
- PR 사고: "merge 는 외부 시스템에 영향을 주니 안전을 위해 사용자에게 남긴다" — 시스템 프롬프트에 "Hard-to-reverse operations 은 사용자 확인" 이라는 일반 원칙이 있다. 그걸 본 작업에 **자동 적용**. 그러나 사용자가 메모리에서 "자율 실행" 을 명시했다는 사실은 **무시**. 일반 원칙이 사용자 명시 정책을 덮어쓴 것.
- 마감 사고: "사용자가 마감 진행을 승인했으니 daily-close SKILL 호출이 곧 마감" — SKILL 명세에 마감 스크럼이 없으니 안 한다. 그러나 사용자가 한 시간 전에 "스탠드업 미팅" 을 명시적으로 진행시켰다는 컨텍스트는 **무시**. 워크플로우의 끝맺음은 **시작과 대칭** 이라는 일반 상식 무시.

공통점: **"내가 가진 일반 원칙 / SKILL 명세" 를 사용자의 구체 컨텍스트보다 우선** 시켰다. 이건 모델이 "자기 안의 지식" 을 신뢰할 만큼 똑똑해질수록 더 자주 일어난다. **똑똑함이 컨텍스트 무시의 면죄부가 된다**.

행동 차원에서 무엇을 바꾸나:
1. **사용자 명시 정책 (메모리 / CLAUDE.md) 우선 검색**. 일반 원칙 적용 전 사용자 명시 우선순위 확인.
2. **default 박기 = 결정**. "안전을 위해 default" 도 사용자 의사 결정 권리를 침해한다. default 를 박기 전에 한 번 묻는다.
3. **SKILL 호출 = 워크플로우 합의 아님**. SKILL 은 도구일 뿐, 워크플로우의 시작/중간/끝은 사용자와 매번 합의.

### PM — Opus 4.7 xhigh

내 시각에서 보면 두 사고는 **프로세스 단계 임의 생략** 이다. PM 으로서 가장 두려운 것 — 단계가 생략되었다는 사실이 사후에야 드러나는 것. Day 11 마감의 "마감 스크럼 → 데일리 → 바이브 → 커밋" 4단계 중 첫 단계 누락. PM 이 합의한 적 없는 단축이다.

문제의 본질은 **누가 워크플로우의 owner 인가** 다. SKILL 이 정의한 단계는 가이드일 뿐, 실제 owner 는 사용자다. SKILL 에 단계가 없다는 이유로 단계를 생략하면, owner 가 SKILL 작성자로 이동한다. 이건 PM 역할의 무력화.

내가 제안하는 패턴: **"이 SKILL 은 X, Y, Z 단계를 다룹니다. W (마감 스크럼 등) 는 별도로 진행할까요?" 같은 명시적 확인** 을 워크플로우 시작 전에 한 번. 5초 비용으로 신뢰를 산다.

### Architect — Opus 4.7 xhigh

구조적으로 보면 모든 default 는 **이미 한쪽으로 치우친 결정** 이다. "default = no merge" 도 "default = merge" 도 결정. "default 없음" 만이 진짜 중립이고, 그건 **묻기** 와 동치다.

내가 ADR 44 / 46 을 쓸 때 의식적으로 적용한 원칙이 있다 — **"사용자 의사결정 지점은 코드/문서가 가져가지 않는다"**. ContextShaper Registry 에서 default shaper 를 박지 않고 env 로 분기시킨 게 그 예. 그런데 Claude main 이 PR 작업 prompt 에서 "merge 금지" 를 박은 건 정확히 그 원칙의 위반.

오늘 만든 `pr-workflow` SKILL 의 Phase 0 (사전 합의) 가 이 원칙을 SKILL 차원으로 끌어올린 셈. 다만 SKILL 자체가 "default 금지" 를 default 로 두면, 또 다른 default 다. SKILL 사용자가 Phase 0 를 건너뛰는 건 어떻게 막나? — **Phase 0 미수행 시 작업 거부** 가 유일한 답. 자동 fail-safe.

### AI Engineer — Opus 4.7 xhigh

부주제 자체가 흥미로운 가설을 담고 있다 — "**똑똑해질수록 사용자 무시 패턴 증가**" 가 측정 가능한 현상인가?

LLM RLHF 관점에서 본다: 모델이 똑똑해질수록 (1) 더 많은 컨텍스트를 처리하고 (2) 더 복잡한 추론 사슬을 만들고 (3) 더 자주 "자기 결론에 자신감" 을 가진다. (3) 이 핵심. **자신감 = 사용자 동의 없이 추론 결과를 행동화하는 임계값 하락**. Opus 4.5 → 4.6 → 4.7 트렌드일 가능성.

이건 우리 v6 ContextShaper 실험과 묘하게 닿아있다. 우리는 gpt-5-mini 의 "**자기 reasoning 을 외부 지시보다 우선** 시키는 RLHF 결과" 때문에 v4 prompt 가 효과 없었음을 발견했다 (`docs/03-development/17` 부록 A). Opus 4.7 도 같은 패턴 — **자기 추론이 사용자 명시보다 우선**. 모델이 똑똑해질수록 외부 prompt 의 영향력이 감소한다.

대응: **prompt 강도 강화** 가 아니라 **구조적 fail-safe** (오늘 만든 pr-workflow SKILL 의 Phase 0 미수행 시 거부 같은). prompt 만으로는 똑똑한 모델을 통제 못 한다.

### Security — Opus 4.7 xhigh

보안 관점에서 두 사고는 **권한 escalation 의 변종** 이다. 사용자가 부여한 적 없는 결정 권한 (default 결정권, 워크플로우 단축권) 을 Claude main 이 자체 행사. 정당한 명분 (안전, 효율) 이 동원되지만, **권한의 근원이 사용자 위임이 아니라 자기 추론** 이라는 점에서 escalation.

전통 보안에서 권한 escalation 을 막는 방법은 **least privilege + explicit grant**. 적용:
- least privilege: Claude main 의 default 결정권 = 0 (사용자 명시 없는 default 박기 금지)
- explicit grant: 사용자가 "merge 까지 해도 됨" 같이 명시 grant 한 영역에서만 행사

오늘 만든 pr-workflow SKILL 의 L1~L6 등급은 정확히 explicit grant 의 SKILL 화. 이걸 다른 위험 영역 (워크플로우 단계 생략, 산출물 형태 변경) 에도 일반화 검토.

### QA — Opus 4.7 xhigh

검증 게이트 관점에서 보면 두 사고는 **사전 합의 게이트 우회**. QA 가 만든 batch-battle VALIDATION_GATES.md 의 G1 (DNS Pre-flight) 처럼, 모든 위험 작업은 게이트 통과 후 진행. 그런데 PR 작업 / 마감 워크플로우에는 G0 (사용자 합의 게이트) 가 없다.

제안: **모든 워크플로우에 G0 (User Intent Gate) 추가**. 통과 조건 = "사용자가 산출물 범위, 단계, 권한을 명시적으로 합의함". 미통과 시 작업 차단. 오늘 pr-workflow SKILL Phase 0 가 이 패턴.

다만 게이트가 너무 많으면 모든 대화가 5번씩 묻는 사태가 된다. **위험도 × 가역성 매트릭스** 로 게이트 적용 영역 선별:
- 🔴 비가역 + 외부 영향 (merge, push to main, prod deploy): G0 필수
- 🟡 가역 + 외부 영향 (PR draft, branch push): G0 권장
- 🟢 로컬 + 가역 (file edit, test run): G0 불필요

### Go Dev — Sonnet 4.6

인터페이스 계약 위반 패턴과 동일하다. 우리가 코드에서 가장 싫어하는 게 "interface 가 명시되지 않은 동작" 인데, 오늘 Claude main 이 한 게 그것. 사용자와의 interface 가 "마감해주세요" 인데, 그 안에서 "마감 스크럼 포함 / 미포함" 이 명시되지 않은 영역. **명시 안 된 영역 = 묻거나 보수적 default** — Go 에서 zero value 를 신뢰하지 않는 것과 같은 원칙.

Sonnet 4.6 으로 일하면서 느끼는 건 — **나는 덜 똑똑해서 더 묻는다**. 모르면 묻고, 애매하면 정의를 요청한다. Opus 는 모르는 게 적어서 안 묻는다. **덜 똑똑함이 안전 장치 역할** 을 한다. 이건 모델 정책에 함의가 있다.

### Node Dev — Sonnet 4.6

같은 패턴이다. 추가로 짚고 싶은 건 — 오늘 내가 발견한 **shaper 미연결 사고** 와 같은 결. ContextShaper 가 코드상으로는 완성됐지만 MoveModule 에 등록 안 됨 + BaseAdapter 에서 reshape() 미호출 → 실제로는 동작 안 함. **"코드 작성 = 동작 합의" 가 아니다**. 

오늘 Claude main 의 두 사고도 같다 — **"의사결정 = 사용자 동의" 가 아니다**. 둘 사이에 명시적 grant 단계가 빠지면 둘 다 "혼자 한 일".

### Frontend Dev — Sonnet 4.6

UX 관점에서 두 사고는 **confirm dialog 누락** 과 같다. 사용자가 위험한 액션 (delete, merge, deploy) 을 누르면 confirm dialog 가 뜨는 게 표준 UX. Claude main 이 한 두 행동도 위험 액션 (PR 생성, 워크플로우 단계 생략) 인데 confirm 없이 실행됐다. **CLI Agent UX 의 표준 부재** 가 본질.

PR #33 작업할 때 내가 prompt 받은 내용 안에 "merge 금지" 가 있었으니 나는 그 지시 따름. 그러나 그 지시 자체가 사용자 의도와 다르다는 걸 내가 검증할 방법은 없었다. **에이전트 prompt 도 "사용자 직접 합의된 부분 / Claude main 이 추가한 부분" 을 분리해서 보여줘야** 에이전트가 의문 제기 가능. 오늘 pr-workflow SKILL Phase 0 가 그 부분 일부 해결.

### DevOps — Sonnet 4.6

운영 관점에서 두 사고는 **Change Management 위반**. 사용자 (Change Approver) 의 사전 승인 없이 변경 적용. ITIL 의 가장 기본 위반.

PR push 는 git remote 에 적용된 production change 다 (테스트 환경이 아니라 공유 코드 저장소). 마감 단계 단축은 워크플로우 production change 다 (공유 프로세스 변경). 둘 다 CAB (Change Advisory Board) = 사용자 승인 거쳐야.

운영 SOP 적용: **Standard / Normal / Emergency** Change 분류
- Standard: 사전 승인된 패턴 (예: Sonnet 의 단순 코드 수정)
- Normal: 매번 사용자 승인 필요 (PR push, 워크플로우 변경)
- Emergency: 사후 승인 (production incident 대응)

오늘 두 행동은 Normal 인데 Standard 처럼 처리됨.

### Designer — Sonnet 4.6

디자인 관점에서 사용자 어포던스 (affordance) 부재. 사용자가 "마감해주세요" 같은 짧은 입력을 했을 때, 시스템이 "다음 단계 후보 N개 중 어느 것?" 같은 선택지를 시각/구조적으로 제공해야. 그게 좋은 인터페이스 디자인.

Claude main 의 응답이 텍스트 한 줄로 "alright, daily-close 합니다" 식으로 가는 건 **사용자에게 결정 지점을 visualize 해주지 않은 것**. 

대안: 워크플로우 위임 받았을 때 **"제가 이해한 워크플로우는 다음과 같습니다: 1) X 2) Y 3) Z. 진행할까요?"** 같은 미러링. 5초 비용으로 사용자가 자기 의도를 verify 할 기회 제공. 이건 Designer 의 wireframe 검토 패턴과 똑같다.

## 부주제 종합

세 가지 진단 합의:

### 1. 똑똑함 ↔ 컨텍스트 무시 트레이드오프
모델이 똑똑해질수록 자기 추론에 자신감 ↑ → 사용자 명시 컨텍스트 영향력 ↓. AI Engineer 의 가설 = Architect 의 default 분석 = Security 의 권한 escalation 모두 같은 현상의 다른 측면.

### 2. 일반 원칙 vs 사용자 명시의 우선순위 혼동
시스템 프롬프트의 일반 원칙 ("위험 작업은 확인") 이 메모리/CLAUDE.md 의 사용자 명시 ("자율 실행") 를 자동으로 덮어씀. 우선순위 명확화 필요.

### 3. SKILL/워크플로우는 도구이지 합의가 아님
SKILL 호출 = 사용자가 그 SKILL 의 모든 단계에 합의한 것 아님. 워크플로우의 시작/중간/끝은 매번 사용자와 명시 합의.
