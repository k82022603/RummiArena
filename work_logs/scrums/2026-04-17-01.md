## 논의 사항

### 1. 왜 5명만 Sonnet 으로 내렸나 — 역할과 엔진의 매칭

구현/설정은 정형화된 작업이 많다. Go 의 handler 패턴, NestJS 의 module 구조, Helm values 수정, Next.js 컴포넌트 스캐폴딩 — 이들은 "무엇을 만들지" 가 이미 결정된 후의 작업이다. Sonnet 4.6 이 충분하다.
반대로 architect 의 수정 계획서, ai-engineer 의 프롬프트 설계, security 의 취약점 판단, pm 의 일정 리스크 판단, qa 의 Phase 3 검증 게이트 — 이들은 **틀리면 하위 작업이 전부 잘못된다**. Opus 4.7 xhigh 의 깊이가 필요하다.

### 2. xhigh 는 왜 명시하는가

"opus" 별칭만 쓰면 Anthropic 측 해석이 바뀔 때 의도가 흐려진다. **Extended Thinking High budget** 을 명시하면 — (a) 설계 세션에서 thinking 을 아끼지 말라는 지시가 정책으로 고정되고, (b) 비용 청구서를 봤을 때 "왜 이 세션이 비쌌나" 를 설명할 수 있다. Sonnet 4.6 과 비교되는 순간 xhigh 의 존재가 정당화된다.

### 3. v2-zh 가설 — 언어 오버헤드 제거

R8 reasoning_content 샘플에서 78% 중문 비율이 나왔다. DeepSeek-R1 계열은 중문 코퍼스 비중이 높다. 우리 프롬프트는 영어 → 모델 내부에서 중국어로 재사고 → 영어 응답 — 이중 번역 비용이 latency 와 quality 양쪽을 누를 수 있다.
v2-zh 는 **프롬프트 본문만 중국어로**, tile 코드 (R7a, B13b) 와 응답 포맷은 영어 유지. 단일 변수 (언어) 만 바꾸는 controlled experiment. v2 와 few-shot 구조는 동일.
가설 확증 시 — DeepSeek 전용 production prompt 를 v2-zh 로 전환. 반증 시 — v2 복귀 + "번역 오버헤드는 signal 이 아니었다" 결론.

### 4. Agent Model Policy 의 이력 관리 원칙

변경은 두 곳에 기록한다:
- `.claude/agents/{name}-agent.md` frontmatter `model:` 필드 **주석**
- `CLAUDE.md` §Agent Model Policy §이력
두 곳이 같은 이야기를 하므로 한쪽만 바뀌면 QA 가 규칙 위반으로 잡아야 한다 — 추가 QA 체크 항목.

### 5. 논문 초안 시점

v2-zh 까지 끝내면 dataset 이 "v1 baseline / v2 few-shot / v3 tuned / v4 reasoning / v4.1 thinking-budget-removed / v5 zero-shot / v5.1 reinforced / v5.2 hybrid / v2-zh" 9종이 된다. Sprint 6 내에서 논문 초안 착수가 가능한 분량. **Day 7 저녁에 v2-zh 결과 보고 GO/No-Go 판단** 하기로.
