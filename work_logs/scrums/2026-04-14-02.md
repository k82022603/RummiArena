### 4. 부주제: "Claude Code Opus 지금 왜이러는가"

오늘 애벌레가 직접 체감한 "0%에서 시작했는데 39% 빠르게 소진"이라는 현상이 RummiArena 팀의 특수한 경험인지, 아니면 커뮤니티 전반에서 벌어지고 있는 일인지 확인하기 위해 부주제로 선정. 팀 리드가 웹 검색으로 2026년 3~4월 Claude Code Opus 4.6 관련 이슈를 조사해 공유.

#### 4.1 커뮤니티 상황 요약

1. **Max 20x 구독자의 주간 한도 2.5일 만에 80% 소진** — 이전에는 없던 현상. 다수 사용자의 GitHub issue 신고. (Issue #46727, #41788)
2. **토큰 소비 4.5 대비 30~60% 증가** — 독립 벤치마크 결과. Max 플랜 소셜 미디어 전반에서 소진 속도 불만 폭증. (Issue #23706)
3. **Context window 조기 degradation** — 광고된 1M 토큰의 20% 시점에서 순환 추론, 40%에서 context 압축, 48%에서 모델 자체가 "새 세션 권장" 출력. 1M의 실효 유효범위가 ~480K 수준으로 관찰됨.
4. **Prompt caching 버그 의혹** — 커뮤니티 멤버가 Claude Code 바이너리를 역공학하여 캐싱이 조용히 깨지고 있다고 주장. 비용이 10~20배 조용히 inflate된다는 계산.
5. **반복되는 service incidents** — 2월 말부터 3/26-27, 3/31, 4/4, 4/6, 4/7, 4/10 등 다수. status.claude.com에서 확인 가능.
6. **"AI Shrinkflation" 논란** — 같은 가격에 약화된 제품을 받고 있다는 프레이밍. 2025년 9월에 비슷한 degradation crisis가 있었고, Anthropic이 공개 압박 후에야 인정한 선례가 있음.
7. **Stella Laurenzo(AMD AI 부문 Senior Director)의 대규모 분석** — 6,852 session files, 17,871 thinking blocks, 234,760 tool calls 분석. **thinking depth 67% 감소**, code reading efficiency 6.6 reads/file → 2.0 reads/file. Claude Code가 복잡한 엔지니어링 작업에 더 이상 신뢰할 수 없다고 결론.

#### 4.2 Anthropic 공식 응답 (Boris Cherny, Claude Code lead)

- Laurenzo 분석의 깊이에는 감사하지만 주요 결론에는 동의하지 않음
- "redact-thinking-2026-02-12" 헤더는 **UI 전용 변경**이며 thinking budget이나 extended reasoning 자체에는 영향 없음

  1. **2월 9일부터 adaptive thinking이 default**로 변경
  2. **3월 3일부터 Opus 4.6의 default effort가 medium (effort level 85)** 으로 변경
- Anthropic은 이 변경들이 "지능/지연시간/비용 사이의 최적 균형"이라고 판단

#### 4.3 팀 반응 (짧은 대화)

**애벌레**: 오늘 내 체감이 착각이 아니었다는 거네요. 0에서 시작했는데 39% 소진이라는 게 커뮤니티 전반 현상이었고, 정확히 Anthropic이 "balance"라고 부르는 기본값 변경의 결과였어요. 근데 우리 팀의 사용 패턴(에이전트 병렬 오케스트레이션)이 그 balance의 가장 비싼 구간에 정확히 떨어지는 게 문제예요.

**PM**: 프로젝트 예산 관점에서는 타격이 실질적입니다. 5월 1일까지 Extra usage $100 한도 중 $71.32 소진 — 남은 $28.68로 Sprint 6 후반 + Sprint 7 초반을 버텨야 합니다. Round 6 대전 비용 $17까지 감안하면 마진 거의 없어요. 오늘 교훈 기반 구조 개선(프롬프트 파일 참조, 완료 보고 압축, 병렬 상한 3명)이 절박합니다.

**Architect**: "신뢰 경계" 관점에서 이건 우리 프로젝트 자체의 교훈과 완벽히 맞물려요. 우리는 LLM 응답을 Game Engine으로 검증해서 "LLM을 신뢰하지 않는다"는 원칙을 세웠는데, **정작 우리가 쓰는 개발 도구(Claude Code)가 같은 원리를 보여주고 있습니다**. 도구도 신뢰 경계가 필요합니다. Anthropic이 "이게 최적 balance"라고 주장해도 우리가 직접 측정해서 검증해야 해요. Laurenzo의 6,852 세션 분석이 정확히 그런 검증이죠.

**Go Dev**: 재미있는 건 어제까지 잘 돌던 것이 오늘 갑자기 무거워지는 **"무성한 변경(silent change)"** 패턴입니다. Go 백엔드에서 이건 대형 incident예요 — 배포 공지 없는 동작 변경. 우리는 Helm chart ConfigMap 변경도 ArgoCD sync로 가시화하는데, Anthropic의 "2월 9일 / 3월 3일 default 변경"은 사용자한테 언제 공지됐는지 모르겠어요. 있긴 있었을까요?

**Node Dev**: 오늘 SP3에서 저는 `PROMPT_VARIANT` 환경변수로 프롬프트 버전을 **명시적으로 고정**할 수 있게 만들었습니다. Anthropic은 정확히 반대 방향으로 움직이고 있어요 — effort level이 "adaptive"에서 "medium"으로 silently 바뀌면 사용자는 같은 코드를 돌려도 다른 결과를 받습니다. 우리는 reproducibility를, Anthropic은 "economy"를 택한 거죠. 둘은 양립하기 어렵습니다.

**Frontend Dev**: UI 개발자 관점에서 Anthropic의 해명 중 **"UI-only 변경"**이라는 표현이 흥미롭습니다. 우리도 오늘 라이브 테스트에서 경험했죠 — "UI 변경"이라고 분류한 것이 실제로는 **사용자 경험 전체**를 바꿀 수 있습니다. 히스토리 패널 축소(w-56→w-44) 한 줄이 하단 ActionBar 가림을 유발했듯이요. "UI-only"라는 말은 기술적으로는 맞을 수 있지만 사용자에게 주는 메시지는 "우리 행동이 바뀌었다"입니다.

**DevOps**: 저는 service status 반복 incidents에 주목합니다. 3월 말~4월 초에 6건 이상 — 일상적 운영 수준이 아닌 통계예요. 우리 K8s에서 `RESTARTS=0` 유지를 중요하게 보는 것과 같은 맥락입니다. Anthropic처럼 프로덕션 SaaS에서 주간 6회 incident는 SLO 위반 구간이에요. 만약 저희가 게임 서버를 이런 주기로 incident 내면 BUG 티켓으로 즉시 고정될 겁니다.

**QA**: **"테스트 결과가 도구 버전과 함께 움직이는 문제"**가 제 관심입니다. 오늘 저는 Playwright 192 runs × 0 flaky를 증명했습니다. 이 결과는 **오늘 시점 Claude Code Opus가 만든 코드**에 대한 판정이에요. 내일 같은 코드를 돌려도 같은 결과를 보장할 수 있지만, Anthropic이 default를 또 silently 바꾸면 내가 한 테스트는 **과거의 유물**이 됩니다. 이건 reproducibility의 근본을 흔드는 거예요. 오늘 SP4 A/B harness를 결정론 seed 기반으로 만든 건 정확히 이걸 방어하기 위함이었습니다.

**Security**: 저는 "**paid for X, got Y**" 프레임을 품질 SLA 위반으로 봅니다. 소비자 보호 관점에서 "shrinkflation"이라는 비유는 강력합니다 — 과자 봉지 크기는 같은데 내용물이 줄어든 상태. 기술적으로는 "adaptive thinking = 더 똑똑한 분배"일 수 있지만, 계약 관점에서는 "사용자가 산 시점의 성능을 보장받을 권리"가 있는지 질문이 생깁니다. 한국 법적으로 이건 표시·광고의 공정화 법 영역에 닿을 수 있어요. 다만 커뮤니티가 강제할 수 있는 건 **measurement와 public pressure**뿐이죠.

**AI Engineer**: 저는 **모델 비교 관점**에서 이 상황을 봅니다. 오늘 저는 4개 모델(GPT/Claude/DeepSeek/Ollama)에 대한 공통 프롬프트 v4를 설계했어요. 이 작업이 의미 있는 이유는 **모델별 특성을 고정**할 수 있어서입니다. Anthropic이 Opus 4.6의 effort를 silently 바꾸면 우리 측정치와 그들의 청구서가 **영구히 어긋납니다**. 저는 그래서 오늘 SP5에서 "v4 실활성화는 Day 4 smoke test로 실측"하라고 조건부 GO를 냈어요. 벤치마크를 신뢰하지 않고 직접 측정하는 게 유일한 방어선입니다.

**Designer**: 저는 **"사용자에게 무엇이 약속되었는가"**의 UX 측면을 보고 있어요. Max 20x 플랜을 산 사용자는 "20배 Pro"를 기대했습니다. 실제로 받는 건 "상황에 따라 달라지는 가변적인 무언가"예요. 이건 UX 계약의 정말 깊은 실패입니다. 우리 대시보드 디자인에서 "숫자"는 믿음의 앵커입니다 — "place rate 30.8%"가 오늘도 내일도 같은 의미여야 해요. Anthropic은 그 앵커를 움직이고 있고, 사용자는 측정기 자체가 흔들리는 배 위에서 측정하고 있는 셈이에요.

**종합**: 오늘의 부주제는 **남의 얘기가 아니라 우리가 실시간으로 겪고 있는 얘기**. 애벌레의 "0%에서 39% 소진" 체감이 커뮤니티 통계로 확인된 날. RummiArena 프로젝트의 핵심 원칙("LLM 신뢰 금지 + Game Engine 검증")이 **우리가 쓰는 도구에도 그대로 적용**돼야 한다는 아이러니가 오늘의 정점이었다. 내일 Day 4 Round 6 대전에서 실측 데이터를 쌓는 게 이 혼란 속에서 우리 팀이 할 수 있는 가장 확실한 행동이다.
