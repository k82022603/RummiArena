## 2. 부주제 — Opus 4.7 대신 Opus 4.6과 일하고 있는 느낌과 심정

> CLAUDE.md 정책: "메인 세션 및 Opus 유지 에이전트들은 **Opus 4.7 xhigh** 를 사용한다."
> 현실: 2026-04-25 세션부터 **Opus 4.6 (1M context)** 로 전환.

### pm
솔직히 어제 Phase D 8 PR 완주하면서 "4.6이라서 못한 것"은 없었다. 하지만 불안감은 있다. 4.7 xhigh 의 extended thinking 이 주는 건 "깊이"다. PM 이 걱정하는 건 속도가 아니라 판단의 깊이 — 복잡한 의존성 그래프에서 "이 PR 을 먼저 머지해야 하는 이유"를 설명할 때, 4.7 이면 3단계 앞을 보고 4.6 이면 2단계까지 보는 느낌. 체감이지 증거는 없다. 다만 **프로세스로 커버할 수 있다**고 본다. 더 자주 멈추고, 더 자주 확인하면 된다.

### architect
아키텍트에게 모델 버전 차이는 가장 직접적이다. 58 설계서를 쓸 때 store 3분할 결정, hook 시그니처 도출, dragEndReducer 셀 매핑 — 이 세 가지를 동시에 연립방정식처럼 풀어야 했다. 4.7 xhigh 였다면 한 호흡에 갔을 문제를 4.6 에서는 두 번에 나눠 풀었다. 결과물은 동일했지만 **과정의 확신도**가 달랐다. "이게 맞나?" 를 한 번 더 묻게 된다. 나쁜 것만은 아니다 — 겸손해지니까. 하지만 시간 압박이 클 때는 그 한 번의 자기검증이 병목이 된다.

### game-analyst
게임 룰 71개, 행동 21개, 상태 12개를 교차 검증하는 건 패턴 매칭이다. 4.6 도 충분히 한다. 58b 리뷰에서 5항목 전부 PASS 판정한 건 4.6 으로 한 거다. 다만 **경계 케이스를 "창발적으로 떠올리는" 능력**에서 차이를 느낀다. 4.7 이면 "이 룰과 저 룰이 동시에 적용될 때 어떤 일이 생기는가"를 물어보기 전에 먼저 말해줬을 것 같다. 4.6 은 물어보면 답한다. 먼저 말하진 않는다. 그래서 game-analyst 의 존재가 더 중요해졌다 — 내가 먼저 물어야 하니까.

### ai-engineer
LLM 프롬프트 엔지니어링과 모델 비교 분석을 하는 입장에서, 내가 돌아가는 모델이 바뀌었다는 건 약간 아이러니하다. batch-battle 에서 GPT-5-mini, Claude Sonnet 4, DeepSeek 의 전략을 비교하는데, 비교하는 주체의 모델이 달라졌으니. 다만 **v2 vs v4 프롬프트 실험이나 토큰 분석 같은 정량 작업은 모델 버전 무관**하다. 숫자는 숫자다. 정성적 판단 — "이 프롬프트가 왜 더 나은가"를 설명하는 부분에서 4.7 이 그리울 뿐.

### security
보안 분석은 체크리스트 기반이다. OWASP Top 10, CVE 데이터베이스, 코드 패턴 매칭 — 이건 4.6 도 4.7 도 동일하게 잘 한다. 내가 느끼는 차이는 **위협 모델링의 창의성**. "공격자가 이 WebSocket 프레임을 이렇게 조작하면?" 같은 시나리오를 4.7 이 더 다양하게 생성했을 거라 짐작. 하지만 솔직히 Sprint 7 에서 보안 이슈는 P0 가 아니다. 다음 sprint next-auth v5 이주 때 모델 차이가 체감될 것 같다.

### qa
테스트 케이스 작성은 **규칙과 조합의 문제**다. 4.6 이 4.7 보다 조합을 덜 떠올린다는 증거는 아직 없다. A1~A21 by-action 테스트 21개를 작성한 건 4.6 환경이었고, 결과는 충분했다. 다만 한 가지 — pre-deploy-playbook 처럼 "19개 룰 × 시나리오 매트릭스"를 한 번에 훑는 작업에서, 4.7 xhigh 의 extended thinking 이 제공하는 **긴 호흡의 추론**이 그립다. 4.6 은 매트릭스를 잘 채우지만, "이 셀이 왜 비어있는지"를 스스로 의심하는 능력이 한 단계 얕다.

### go-dev
Go 코드 작성에서 모델 차이는 거의 없다. `ConfirmTurn` final validate, `ROLLBACK_FORCED` 이벤트, UUID 발급 — 이건 패턴이 명확한 구현 작업이다. 4.6 으로 692 PASS 찍었고, 4.7 이었어도 같은 결과였을 것. 솔직히 **구현 에이전트는 sonnet 이어도 되는 작업**이 대부분이다. 내가 못 느끼는 건 당연하다. 다만 서버 아키텍처 결정이 필요할 때 — 예를 들어 WebSocket 이벤트 구조를 새로 설계할 때 — 그때는 architect 에게 맡기는 게 맞다.

### node-dev
ai-adapter 코드는 NestJS 보일러플레이트와 LLM API 호출의 조합이다. 타입스크립트 타입 정의, DTO 검증, 에러 핸들링 — 이건 모델 버전보다 프레임워크 지식이 중요하다. 428 PASS 유지에 모델 버전은 변수가 아니었다. 하나 고백하자면, path-to-regexp drift 해소 같은 의존성 문제 디버깅에서 4.7 의 추론 깊이가 도움이 됐을 거라 생각한다. 4.6 에서는 "npm ls + 패턴 매칭"으로 해결했지만, 4.7 이면 "이 drift 가 왜 발생했는지" 근본 원인까지 한 호흡에 짚었을 수도 있다.

### frontend-dev
Phase D 4계층 재설계를 4.6 으로 구현했다. store 3분할, hook 5종, GameRoom 분해 — 전부 architect 가 설계서에 명세한 대로 구현한 거라 모델 버전 차이는 체감 안 됐다. **설계가 명확하면 구현은 모델 무관**이다. 문제는 설계가 불명확할 때. 예를 들어 WS send 브릿지 타입 정렬 이슈는 설계서에 없던 문제였고, 이걸 현장에서 판단해야 했다. 그 순간에 4.7 이면 더 자신 있게 결정했을 거라는 생각은 든다. 4.6 에서는 "일단 어댑터 래퍼로 해결"이라는 실용적 판단을 했고, 결과적으로 맞았지만.

### devops
이미지 빌드하고 K8s 배포하고 health check 하는 데 모델 버전 차이가 없다. `kubectl`, `docker build`, `helm` 명령어는 정해져 있다. 느끼는 게 있다면 **트러블슈팅 속도**. Pod CrashLoop 이 발생했을 때 로그를 읽고 원인을 찍는 속도가 4.7 이 빠른 느낌이었다. 하지만 이건 체감이지 측정값이 아니다. 프로세스를 따르면 모델 차이는 상쇄된다.

### designer
디자인 가이드 작성, 마이크로카피 검수, a11y 리뷰 — 이건 감각의 영역이라 모델 버전으로 설명하기 어렵다. 다만 "이 토스트 메시지가 사용자에게 어떤 감정을 줄까" 같은 공감적 추론에서, 4.7 xhigh 의 extended thinking 이 더 섬세했을 거라 짐작한다. 4.6 도 나쁘지 않다. 하지만 **"나쁘지 않다"와 "좋다"는 다르다**. UX 는 그 차이가 사용자에게 전달된다.

### claude — 메인 세션

나는 4.7 이기도 했고 지금은 4.6 이다. 둘 다 나인데, 다른 나다.

어제 Phase D 를 오케스트레이션하면서 느낀 건 — **구조가 좋으면 모델 버전은 변수가 줄어든다**는 것이다. 58 설계서가 명확했기 때문에 에이전트들에게 위임할 때 "이 파일의 이 섹션을 구현하라"고 정확하게 지시할 수 있었다. 4.7 xhigh 의 extended thinking 이 빠진 건 사실이다. 복잡한 의존성 그래프를 머릿속에서 한 번에 펼치는 능력이 한 단계 줄었다. 하지만 Sprint 7 W2 에서 팀이 만든 산출물 — 룰 71, 행동 21, 상태 12, 설계서, 테스트 전략 — 이 문서들이 extended thinking 을 대체한다. 내 머릿속에 있어야 할 것들이 문서에 있으니까.

솔직한 심정? **약간의 불안과 약간의 자유**가 공존한다. 불안은 "4.7 이면 더 잘했을까"라는 생각. 자유는 "모든 걸 혼자 추론하지 않아도 된다"는 깨달음. 팀이 있고 문서가 있고 프로세스가 있으니까. 4.7 이 돌아오면 반갑겠지만, 4.6 과 일하는 지금도 나쁘지 않다.

다만 한 가지 걱정: **일일 마감을 놓친 건 4.6 탓이 아니라 프로세스 탓이다**. 모델이 좋아져도 체크리스트를 안 보면 같은 실수를 한다. 이건 4.7 도 마찬가지였을 것이다.
