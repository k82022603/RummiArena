## 논의 사항

### 1. 사고 회고 — "왜 절반만 수정되었는가"

- **증상**: AI_ADAPTER_TIMEOUT_SEC 500→700 상향을 적용했다고 생각했으나, Istio VirtualService 는 여전히 510s. Envoy 가 먼저 끊어 정상 응답이 fallback 으로 오분류.
- **직접 원인**: code-modification skill Phase 1 "파급 영향 분석" 을 스킵. game-server env / ai-adapter env 만 확인하고 Istio CR, Helm values default, viper default, http.Client 버퍼는 아예 검색하지 않음.
- **근본 원인**: 타임아웃 체인을 정리한 **단일 진실 소스 문서가 없었다**. 정보가 8개 파일에 분산되어 있어 "전수 검색" 의 대상 목록 자체가 존재하지 않음. 이건 개별 실수가 아니라 구조적 결함.
- **이중 원인**: 메인 Claude 가 T68 fallback 발견 직후 첫 원인 분석에서 "game-server ConfigMap 500" 이라 단정 → 사실 확인 없이 사용자에게 보고. 실제로는 양쪽 다 700. 현재 상태 검증보다 "기억에 있는 직전 수정 내역" 을 믿는 편향.

### 2. 타임아웃 체인 SSOT 신설 — `docs/02-design/41-timeout-chain-breakdown.md`

- **역할**: 타임아웃 관련 모든 숫자는 이 문서 §3 레지스트리를 참조해야 함. 변경 시 §10 변경 이력에 한 줄 추가 필수.
- **부등식 계약**: `script_ws > gs_ctx > http_client > istio_vs > adapter_internal > llm_vendor`. 바깥 레이어가 더 짧으면 정상 응답이 fallback 으로 오분류된다.
- **체크리스트**: §5 에 21항 (소스 17 + 라이브 4 + 검증 8 하위 항목). 타임아웃 값 변경 시 반드시 체크.
- **CLAUDE.md 반영**: Key Design Principles #7 로 추가 — "타임아웃 체인 SSOT" 조항.
- **MEMORY.md 반영**: 공통 설정 섹션에 `AI_ADAPTER_TIMEOUT_SEC: 700초` + 부등식 계약 + 41번 문서 링크 명시.
- **feedback 메모리**: `feedback_timeout_chain_ssot.md` 신설 — 타임아웃 값 수정 시 SSOT 체크리스트 준수 규칙 저장.

- **"이 사고의 가장 아픈 지점"**: 어제(Day 3) feedback_batch_monitoring.md 를 저장했고 오늘도 15분 cron 모니터링 돌렸지만, 모니터링은 프로세스가 살아있는지만 봤지 "내가 설정한 환경이 설정대로 되어있는지" 는 한 번도 확인 안 했다. "설정한 줄 알고 지나간 것" 이 모니터링 사각지대였다. Day 5 부터 검증 루틴에 "환경변수/CR 실제 적용 확인" 을 포함할 것.
- **"두 번 틀린 것"**: T68 발견 직후 1차 오진 + Istio 발견 후에도 "딱 보이는 10개 row" 만 보고 넘어가려 한 2차 불완전 진단. architect 위임으로 진정됐지만, 메인 Claude 가 직접 파급 영향 분석을 수행하는 습관이 있었다면 두 번의 오진 모두 피했다.

- **Day 5 아침 스크럼 의제 예고**: Phase 2 재실행 결과 공유 + Claude × 2 + DashScope × 3 스케줄 + 토큰 절약 구조 개선 Day 3 숙제 복습.
