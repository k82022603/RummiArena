#### 3.1 사용자 제공 스니펫의 치명적 오류 — 반대로 해석된 `DISABLE_ADAPTIVE_THINKING`

어제 공유된 설정:
```json
{
  "env": {
    "CLAUDE_CODE_EFFORT_LEVEL": "max",
    "CLAUDE_CODE_DISABLE_ADAPTIVE_THINKING": "1",
    "CLAUDE_CODE_DISABLE_1M_CONTEXT": "1"
  }
}
```

해설 주석은 이렇게 적혀 있었다:
- `EFFORT_LEVEL=max`: 매 턴 풀 추론 강제
- `DISABLE_ADAPTIVE_THINKING=1`: 생각 예산 자체 축소 방지
- `DISABLE_1M_CONTEXT=1`: 컨텍스트 200k 제한, 추론에 집중

**오늘 아침 웹 검색 결과 (공식 Claude API Docs + Jaime Medina X 검증):**

1. **`CLAUDE_CODE_DISABLE_ADAPTIVE_THINKING=1`은 "축소 방지"가 아니다.** 공식 문서 원문: *"revert to the previous fixed thinking budget, controlled by MAX_THINKING_TOKENS"*. 즉 adaptive reasoning을 **끄고 고정 예산으로 회귀**시키는 플래그. 오늘 아침 기본값(adaptive high)보다 **더 적은 생각 예산**이 될 수도 있다. 어제 스니펫의 "축소 방지"라는 주석은 반대로 해석한 것
2. **Effort는 3단계가 아니라 4단계**: `low` / `medium` / `high` / **`max`**. `max`는 thinking token 상한이 없어 가장 느리고 가장 비싸다. 스니펫의 `EFFORT_LEVEL=max` 설정은 실존하지만 이걸 켜면 **토큰 한도 소진을 가속하는 루트**가 된다 (한도 절약 목적으로 켠 거면 역효과)
3. **`DISABLE_1M_CONTEXT=1`은 컨텍스트를 200k로 제한**하는 것은 맞음. 하지만 이건 RummiArena 프로젝트처럼 긴 문서 참조가 많은 워크로드에는 오히려 독이 될 수 있다 (프로젝트 파일 전수 읽기 불가)

결론: **어제 커뮤니티에서 유행하던 "settings.json 치트코드"는 절반만 맞다.** 우리 팀은 적용하지 않는 쪽으로 결정한다.

#### 3.2 우리가 어제 몰랐던 최신 변화 2건

**(A) 2026-04-07: API default effort 재변경 (medium → high)**
- Anthropic이 API/Bedrock/Vertex/Foundry/Team/Enterprise 사용자 default effort를 **medium에서 high로 되돌림**
- 어제 바이브 로그는 "2026-03-03 default medium 전환"까지만 기록 — 지금(4/15)은 **8일 전에 또 한 번 움직였다**는 뜻
- 즉 2월 9일 adaptive thinking default / 3월 3일 medium 전환 / **4월 7일 high 전환** — 두 달 사이 세 번의 silent change
- Go Dev가 어제 지적한 "Helm ConfigMap도 ArgoCD sync로 가시화하는데"의 논점이 **정확히 오늘 다시 유효**

**(B) 2026-04-09: Claude Platform Advisor Strategy 공식 통합**
- Anthropic이 **"Opus=조언자 + Sonnet/Haiku=실행자"** 페어링을 플랫폼에 공식 지원한다고 발표
- "near Opus-level intelligence at significantly lower cost" 포지셔닝
- Explore 서브에이전트는 이미 Haiku 기본 (기존 사실)
- **우리 팀의 어제 오후 개선안 "날 성격별 모델 선택"이 정확히 이 Advisor Strategy 방향과 일치**

#### 3.3 팀원별 관점 한 마디 (12명 순서)

**애벌레**: 어제 내가 공유한 스니펫이 반대로 해석된 거였다니 좀 뜨끔하네요. 내가 원본 글(threads.com)을 검증 없이 가져왔고, 팀 리드도 어제는 그걸 "사용자 제공 맥락"으로 받아 넘겼고. 오늘 아침에 교차 검증으로 오류가 드러난 게 다행입니다. 이거 자체가 우리 프로젝트 원칙의 미니어처예요 — **LLM 답변도, 커뮤니티 합의도, 내가 보낸 스니펫도, 전부 검증 대상**. 오늘 대전도 같은 태도로 갑니다.

**PM**: 4월 7일 재변경이 특히 PM 관점에서 아프네요. 우리 Sprint 6 일정은 "4월 7일 이전 토큰 소비율" 가정으로 잡혀 있었거든요. 8일 사이 Anthropic의 default가 바뀌어서 우리 실제 소비는 더 빨리 쌓였을 수 있어요. 어제 $71.32/$100 Extra usage도 이 변경의 간접 영향일 수 있고. **Sprint 백로그에 "외부 도구 변경 모니터링"을 상시 칸반으로 추가합니다**. Advisor Strategy 공식 발표는 우리 Agent Teams 템플릿 재설계에 공식 근거가 돼요 — 더 이상 "내 판단으로 Sonnet 써볼게요"가 아니라 "Anthropic이 공식 권고한 패턴"으로 주장할 수 있습니다.

**Architect**: "외부 도구의 silent change를 탐지할 것인가"가 **새 ADR 후보**로 올라왔어요. 우리는 ArgoCD로 내부 배포를 가시화하는데, Anthropic은 배포 공지 없이 default를 세 번 바꿨습니다. 이 비대칭을 해결하려면 우리 쪽에서 **effort level 측정 로그**를 직접 남겨야 해요 — 예를 들어 매 Agent Teams 세션 시작 시점에 `claude --version` + `claude config get effortLevel` 같은 정보를 세션 로그에 자동 기록하는 식. 어제 node-dev가 만든 `PROMPT_VARIANT` 로깅과 정확히 같은 패턴을 도구 레이어에도 적용하는 겁니다.

**Go Dev**: 제가 어제 말한 "silent change" 지적에 오늘 4/7 재변경이 결정적 증거로 붙었네요. 두 달에 세 번이면 이건 **"실험적 플랫폼" 모드**로 봐야 합니다. 프로덕션 가정을 깔고 일하면 안 되는 상태예요. Go 백엔드에서 우리가 `GOGC=50` 같은 파라미터를 명시적으로 고정하는 것처럼, Claude Code도 `effortLevel`을 **프로젝트 settings.json에 명시 고정**해야 해요 — Anthropic이 뭘 바꾸든 우리 프로젝트는 영향 없게. 오늘 중 settings.json에 `"effortLevel": "high"` 명시 검토 제안합니다.

**Node Dev**: 어제 스니펫의 `DISABLE_ADAPTIVE_THINKING=1`이 반대로 해석됐다는 건 저한테도 교훈이에요. **SP3에서 저는 `PROMPT_VARIANT` 환경변수를 만들면서 README에 정확한 동작을 주석으로 박아놨어요** — "이 환경변수를 설정하면 해당 어댑터는 명시적으로 이 variant를 load합니다" 이런 식으로. 커뮤니티 스니펫이 주석을 반대로 쓴 것이 바로 제가 피하려 했던 함정입니다. **환경변수 문서는 환경변수 자체와 같이 배포돼야** 해요. 오늘 Advisor Strategy 공식 발표를 계기로, 우리 ai-adapter도 "모델 선택 환경변수"를 명시화하는 게 어떨지 검토 — 예를 들어 `CLAUDE_MODEL=claude-sonnet-4-6` 같은 식으로 배치 작업별 모델을 전환 가능하게.

**Frontend Dev**: 어제 부주제에서 제가 "UI-only 변경"이라는 말이 기술적으로는 맞아도 사용자 경험 전체를 바꿀 수 있다고 했잖아요. 오늘 새 증거(4/7 재변경)가 정확히 그걸 보여줍니다 — Anthropic 입장에선 "default 값 변경"이라는 한 줄 설정 변경일 뿐이지만, 우리 입장에선 **오늘 같은 코드를 돌려도 어제와 다른 결과**를 받는 것. 프론트엔드 버그 다루듯이 이것도 "regression"으로 분류하고 수정본이 올 때까지 우리 쪽에서 workaround를 유지해야 해요. 우리 workaround는 **`effortLevel` 명시 고정**이 되겠네요.

**DevOps**: Architect가 제안한 "세션 시작 시점 effort level 자동 기록"을 제가 구현할 수 있어요. `scripts/agent-team-prelude.sh` 같은 스크립트에 `claude config get effortLevel`, `claude --version`, `date -Iseconds` 기록을 넣고 Agent Teams 가동 스크립트가 이걸 먼저 실행하도록. 로그는 `work_logs/tool-state/` 폴더에 쌓아두면 됩니다. 오늘 Day 4는 대전이 메인이라 Agent Teams 가동이 없으니, **D+1 즈음 조용히 추가**하는 게 맞을 것 같네요. 어제 `rule-matrix-check` CI job 추가한 거랑 같은 패턴으로요.

**QA**: 저한테는 **"reproducibility vs silent change"**가 오늘 가장 큰 주제에요. 제가 어제 증명한 588 runs × 0 flaky는 어제 시점의 Claude Code가 만든 코드에 대한 판정인데, Anthropic이 또 default를 바꾸면 제 테스트는 역사 유물이 됩니다. 4월 7일 재변경이 있었다는 걸 오늘 알았다는 건 **어제 제가 돌린 Playwright 588 runs는 high effort 기반이었다**는 뜻이에요 (4/7 이후). 만약 누군가 내일 이 테스트를 다시 돌려서 다른 결과를 받으면 "flaky"가 아니라 "도구 변경에 의한 variance"일 수 있어요. **테스트 재현 조건 기록**에 "Claude Code effort level"을 추가해야겠네요. 기존에는 "Node 버전, Playwright 버전"만 적었는데.

**Security**: 제 어제 지적("paid for X, got Y")에 Advisor Strategy가 오늘 흥미로운 각도로 닿아요. 4/9에 Anthropic이 공식 발표한 Advisor Strategy는 **사실상 "Opus가 비싸니까 Sonnet/Haiku로 도망가라"는 권고**로 해석할 수 있어요. 원래 Opus로 얻던 지능을 이제 Opus+Sonnet 페어링으로 얻어라 — 이건 **가격 책정 변경 없이 실질 다운그레이드**를 유도하는 패턴입니다. 소비자 보호 관점에서는 "Max 플랜 산 사람은 여전히 Opus 기대한다"는 계약 문제가 남아요. 다만 실용 관점에서는 우리가 이 Advisor 패턴을 적극 활용해서 비용 방어선을 만드는 게 맞고, 그걸 **보안팀 입장에선 "compliance by pattern"**으로 기록해둡니다 — 공식 권고를 따른다는 소명 자료.

**AI Engineer**: 어제 SP1+SP5에서 v4 프롬프트를 만들면서 제가 느꼈던 거랑 오늘 4/7 재변경이 묘하게 맞물려요. 저는 프롬프트 레벨에서 "thinking_budget"을 지시어로 명시화했거든요 — "DeepSeek은 12K 출력까지 사고 허용"이라는 식으로. 그런데 Anthropic은 도구 레벨에서 그걸 silently 바꾸고 있어요. **두 레이어(프롬프트 vs 도구)에서 모두 thinking budget 통제가 필요**하다는 게 오늘의 결론입니다. Round 6에서 Claude 2 게임이 얼마나 "우리 v4 효과 vs Anthropic 4/7 변경"이 섞인 측정일지 주의깊게 봐야 해요. 만약 Claude만 Round 5 대비 눈에 띄게 성능이 다르다면 그건 우리 v4 때문이 아닐 수 있습니다.

**Designer**: 제 어제 발언("Anthropic은 앵커를 움직이고 있고 사용자는 측정기가 흔들리는 배 위에서 측정하고 있다")이 오늘 새 증거로 더 강해졌어요. 4/7, 4/9 두 이벤트가 8일 사이에 연속으로 있었다는 건 **"배가 계속 흔들리는 상태가 뉴노멀"**이라는 뜻. UX 디자인 관점에서 이 상황을 시각화해보자면, RummiArena 대시보드의 metric 카드에 **"측정 시점 Claude Code effort level"**을 미세한 각주로 붙이는 게 어떨까 싶어요. 30.8% place rate라는 숫자 옆에 "measured 2026-04-14, Opus 4.6 effort=high" 같은 메타 각주. 데이터의 앵커를 복원해주는 작은 장치.

**Team Lead**: 제 책임 고백부터 할게요 — 어제 저는 사용자가 공유한 threads.com 스니펫을 받자마자 "저자의 주장"으로 분류하고 넘어갔습니다. 스니펫의 주석이 공식 문서와 반대였다는 걸 어제 검증하지 못했어요. 오늘 아침 제가 다시 파서 발견한 게 그 자체로 24시간 지연입니다. 어제 팀에 공유할 때 "이건 **사용자 제공 비공식 팁**이며, 동작은 별도 검증 필요"라고 명시했어야 했어요. **오늘부터 외부 스니펫 공유 전 공식 문서 교차검증을 기본 동작으로 추가**하겠습니다. 그리고 4/7 재변경과 4/9 Advisor Strategy 공식화는 어제 시점에선 사실상 검색해도 나왔을 정보인데, 제가 첫 검색에서 2026-03-03 medium 전환까지만 확인하고 멈춘 게 한계였어요. **어제의 나는 "오늘까지 있었던 모든 일"을 보지 못했다** — 이게 오늘 배운 개인 교훈입니다.

#### 3.4 팀 결론 (부주제 종합)

- **threads.com 스니펫은 적용하지 않는다** (절반은 반대로 해석된 주석)

- **우리 프로젝트 원칙 "LLM 신뢰 금지 + Engine 검증"이 오늘 한 번 더 도구 레이어에 확장됨** — reproducibility를 위한 측정은 **우리 책임**
