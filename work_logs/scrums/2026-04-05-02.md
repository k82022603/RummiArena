## 각자 공유

### PM

오늘은 Day 4에서 만든 코드를 실전 환경에 올리고 검증하는 날이었다. K8s 재배포를 게이트로 잡고, 이후 서비스 검증 → 대전 + 플레이테스트를 병렬로 돌린 오케스트레이션이 잘 작동했다. 다만 CI/CD에서 7번 파이프라인을 돌린 건 예상 밖. lint-nest, lint-go, build-frontend, scan-admin/frontend — 각각 다른 원인이었고, 하나하나 잡아가면서 최종 17/17 ALL GREEN을 달성했다. 교훈: 에이전트가 만든 코드는 로컬에서 lint를 한 번 돌려보고 커밋해야 한다.

- **블로커**: 없음

### Architect

SEC-RL-003 WS 서버측 메시지 빈도 제한 설계 문서(17-ws-rate-limit-design.md, 837줄)를 완성했다. Fixed Window Counter + In-Memory 방식으로 설계. WS는 연결 단위이므로 Redis RTT가 불필요하고, 글로벌 60 req/min + 타입별 세분화 + 3회 위반 시 Close(4005) 에스컬레이션으로 정리. Go Dev가 Sprint 5 W2에서 바로 구현할 수 있도록 파일 변경 목록(신규 2 + 수정 2)까지 명시. ADR-017로 결정 사유도 기록했다.

- **블로커**: 없음

### Go Dev

K8s 재배포 후 game-server smoke test를 수행했다. Rate Limit middleware가 실제 환경에서 429를 정확히 반환하는지, AI 쿨다운이 Redis TTL 300초로 동작하는지, 비-AI 게임은 쿨다운 영향을 안 받는지 확인. 624개 Go 테스트도 전량 PASS. 오늘은 구현보다 검증에 집중한 날이었다.

- **블로커**: 없음
- **다음**: SEC-RL-003 구현 + BUG-WS-001(TURN_START) 수정

오늘은 직접 투입은 없었지만, DevOps가 발견한 useWebSocket.ts 타입 에러(PONG→PING)가 중요한 이슈였다. C2SMessageType에 PONG이 없는데 비교하고 있었다. 이건 Day 4에서 WS throttle을 추가할 때 PONG(S2C)과 PING(C2S)을 혼동한 실수. 의미적으로도 스로틀 면제 대상은 PING(하트비트)이 맞다.

### DevOps

### Security

오늘은 직접 투입은 없었지만, Architect의 SEC-RL-003 설계를 검토했다. Fixed Window Counter + In-Memory 방식이 WS 메시지 특성에 맞고, 3회 위반 에스컬레이션도 적절. 소스맵 검출 범위 조정(시스템 패키지 제외)은 보안 관점에서도 합리적 — `/usr/local/lib`의 .map은 npm 패키지 소스이므로 역공학 위험이 아님. 잔여 P1(SEC-RL-003)은 Sprint 5 W2에서 Go Dev가 구현 예정.

- **블로커**: 없음

### AI Engineer

두 번 투입. 1차: DeepSeek Round 4 대전 실행. v2 프롬프트로 23.1% Place Rate 달성 — Round 3(12.5%)에서 +10.6pp, A등급. 핵심 발견은 max_tokens 8192→16384 변경: v2 프롬프트가 길어져서 reasoning이 토큰을 모두 소비하면 content가 빈 문자열이 되는 치명적 버그. 2차: 모델별 프롬프트 정책 문서(18-model-prompt-policy.md) 작성. 4종 모델의 전략/파라미터/결과/교훈을 종합 정리.
