오늘의 바이브 로그를 작성/업데이트한다.

## 규칙
- 경로: `work_logs/vibe/YYYY-MM-DD.md`
- 하루 1개. 같은 날짜 파일이 있으면 새 항목 추가, 없으면 신규 생성
- 날짜는 오늘 날짜 자동 적용

## 파일이 없을 때 (신규 생성)
1. 오늘 날짜로 파일 생성
2. 사용자가 공유한 아이디어/영감을 정리하여 작성
3. 사용자에게 검토 요청

## 파일이 있을 때 (업데이트)
1. 기존 파일 읽기
2. 새로운 아이디어를 기존 내용 아래에 `---` 구분선 + 새 섹션으로 추가 (기존 내용 유지)
3. 사용자에게 추가 내용 요약

## 템플릿 (기본)
```markdown
# 바이브 로그

- **날짜**: YYYY-MM-DD
- **주제**: (한 줄 키워드)

## 아이디어 / 영감

(핵심 아이디어를 서술형으로. 무엇이 흥미로운지, 왜 이게 가치 있는지)

## 왜 떠올랐는지

(어떤 맥락/대화/작업에서 이 생각이 나왔는지. 구체적인 트리거)

## 적용 가능성

(이 아이디어를 프로젝트에 어떻게 적용할 수 있는지. 구현 난이도, 실험 설계 가능성)

## 메모

(자유 메모. 관련 기술, 참고할 것, 떠오르는 후속 질문 등)

## 오늘의 한 줄

> (하루를 관통하는 인사이트를 한 문장으로)
```

## 풍부한 바이브 로그를 위한 가이드

### 양보다 깊이
- 아이디어를 나열만 하지 말고, **왜 흥미로운지** 설명한다
- "이건 좋은 아이디어다" 대신 "이건 ~때문에 기존 접근과 다르다"
- 기술적 관찰과 감성적 직관을 모두 담는다

### 다양한 영감 소스
- 작업 중 발견한 패턴이나 우연한 통찰
- 도구 사용 경험에서 얻은 교훈 (예: "ASCII vs Mermaid의 차이", "스킬 공유의 효과")
- 다른 프로젝트/기술/분야에서의 크로스 인사이트
- "만약 ~하면?" 류의 실험 가설
- 실패나 삽질에서 배운 것 (예: "docker MCP가 OOM을 일으킨 이유")

### 구조 활용
- 하루에 여러 영감이 있으면 `---` 구분선 + 별도 `##` 섹션으로 분리
- 각 섹션에 **소제목**을 붙여 나중에 검색하기 쉽게
- 코드 예시, 다이어그램(Mermaid), 비유 등 자유롭게 활용
- **"오늘의 한 줄"**로 하루의 핵심 인사이트를 결정화

### 연결 짓기
- 이전 바이브 로그와 연결되는 아이디어면 언급한다
- "이전에 ~라고 적었는데, 오늘 ~를 해보니 실제로 그랬다"
- 프로젝트의 핵심 가치(LLM 행동 실험, 심리전 시뮬레이션 등)와 어떻게 연결되는지

## 톤
- 자유롭게, 격식 없이
- 학술 논문이 아니라 **연구 노트** 느낌
- 완성된 생각이 아니어도 OK — 반쯤 떠오른 아이디어도 기록할 가치가 있다
- 단, 너무 추상적이면 나중에 읽었을 때 무슨 뜻인지 모르니 **구체적 맥락**은 꼭 남긴다

## 주의사항
- 대화에서 나온 흥미로운 아이디어, 실험 가설, 영감을 놓치지 않고 포착
- 기술적 발견뿐 아니라 프로세스/협업/도구 관련 통찰도 포함
- 여러 주제가 있으면 `---` 구분선으로 섹션 분리
- "오늘의 한 줄"은 선택이지만, 있으면 로그의 밀도가 올라감
