## 1. Executive Summary

### 사건 개요
2026-04-13 사용자 라이브 테스트 중, **루미큐브 핵심 재배치 기능 중 하나인 "합병"(merge)이 드래그-드롭 UI에서 동작하지 않음**을 발견했다. 구체적으로 다음 시나리오가 실패했다:

```
상황: 최초 등록 완료, 테이블에 [R9 B9 K9] 그룹 3장 존재
시도: 랙의 Y9를 [R9 B9 K9] 그룹 위에 드래그-드롭
기대: [R9 B9 Y9 K9] 4색 그룹으로 합병
실제: Y9가 혼자 있는 새 pending 그룹으로 생성됨 (규칙 미구현)
```

### 근본 원인 (한 줄)
`GameClient.tsx handleDragEnd`가 `pendingGroupIds.has(g.id)` 검사로 **이번 턴에 만든 pending 그룹에만** 머지를 허용하고 **서버 확정 그룹은 드롭 대상에서 제외**. 엔진은 해당 동작을 정상 검증하지만 UI가 그 입력을 서버로 전달할 경로가 없음.

### 더 큰 문제
본 감사 결과, **재배치 4유형 전부가 UI/E2E 단계에 빈틈**을 가지고 있음:

| 유형 (§6.2) | Engine 검증 | UI 드래그-드롭 | E2E 테스트 | 상태 |
|------------|-----------|-------------|---------|------|
| 1. 분할 (split) | ✅ V-06 conservation | ❌ 미구현 | ❌ 0건 | **부분** |
| 2. 합병 (merge) — 본 사건 | ✅ V-06 conservation | ⚠️ pending 그룹만 | ❌ 0건 | **버그** |
| 3. 이동 (move) | ✅ V-06 conservation | ❌ 미구현 | ❌ 0건 | **부분** |
| 4. 조커 교체 (joker swap) | ✅ V-07 joker immediate use | ⚠️ 교체만, 회수 후 즉시 사용 검증 미확인 | ❌ 0건 | **부분** |

### 왜 검증 단계에서 안 걸렸나 — 한 줄
**추적성 매트릭스에 "UI" 컬럼 자체가 없었기에** 엔진 검증만 통과해도 "V-13 PASS"로 표기됐고, 이를 "규칙 통과"로 오등가한 것이 근본 결함이다.

| Playtest | ❌ | 0건 |

| Playtest | ❌ | 0건 |

### 3.4 유형 4 — 조커 교체 (Joker Swap)

## 5. 근본 원인 분석

### 5.1 원인 비율 (정직한 추정)
- **지시/계획 불완전 = 60%**
- **검증 프로세스 결함 = 40%**
- **모델 인식 실패 = 부차적 (추적성 매트릭스 PASS 오표기)**
- **Claude Opus 모델 특성 탓 = 아님** (구조적 문제)

### 5.2 세부 인과

#### ① 지시/계획 불완전 (60%)
- Sprint 1 백로그에 "테이블 재배치 지원 | 검증 로직만 (UI는 Sprint 3)" 명시 — **의도는 올바름**
- 그러나 Sprint 3 계획 수립(2026-03-22) 시 이 백로그 항목이 **태스크로 분해되지 않음**
- Sprint 3 "프론트엔드 완성 30 SP"는 4개 이슈(OAuth 8 + 게임보드 8 + WebSocket 8 + gemma3 6)로 구성 → **재배치 UI 태스크 0개**
- Sprint 3 종료 후 어떤 백로그에도 "재배치 UI"가 재등장하지 않음
- **핵심 실패 지점**: "Sprint 1 백로그 → Sprint 3 계획 이월 프로세스"의 부재

#### ② 검증 프로세스 결함 (40%)
- 엔진 단위 테스트와 UI 통합 테스트가 **독립 파이프라인**으로 진행
- "V-13 PASS"는 엔진 `hasInitialMeld` 권한 검증만 통과를 의미했지만, 추적성 매트릭스에서는 그걸 "규칙 통과"로 기록
- Playtest S4는 재배치를 포함했지만, 조커 획득 확률 문제로 **"다음 번에 보완" 상태로 용인** → 재배치 시나리오가 한 번도 실제 실행되지 않음
- 추적성 매트릭스에 **"UI 구현/검증" 컬럼 자체가 없음** → 엔진 통과 = 전체 통과로 오등가

#### ③ 모델 인식 실패 (부차적)
- 2026-04-10 추적성 매트릭스 작성 시 Claude가 엔진과 UI를 구분하지 않고 "PASS"로 표시
- 2026-04-10 에러코드 전수 검토(커밋 `822282e`)에서 "게임룰 3건 구현"을 수행하면서도 재배치 UI 누락을 인지하지 못함
- 다만 이건 **근본 원인이 아니라 계획 단계 누락이 누적된 결과**

### 5.3 왜 Claude Opus 탓이 아닌가
- 모델은 주어진 문서와 백로그를 기반으로 작업
- Sprint 3 계획에 재배치 UI 태스크가 없었다면 모델이 스스로 "빠졌는데?"를 감지하기 어려움
- 추적성 매트릭스 작성 시 **3단계 구분이 없는 템플릿**을 따랐기 때문에 "PASS" 표시가 형식상 정상 동작
- **모델에게 "3단계 매트릭스로 검증하라"고 지시했다면** 누락이 매트릭스 작성 단계에서 발견됐을 가능성이 높음

| 규칙 ID | 규칙 설명 | Engine | Engine 테스트 | UI 구현 | UI 테스트 (E2E) | Playtest | 종합 |
|--------|---------|--------|------------|--------|-------------|---------|------|

## 10. 결론 및 후속 작업

### 사용자 질문에 대한 답
> "게임룰에 따라 제대로 코딩이 되었는지 점검을 했던 것 같은데, 왜 자꾸 코딩이 누락된 것을 사용자 테스트 단계에서나 확인할 수 있는 것일까? Claude Opus 모델 탓일까? 아니면 처음부터 내가 지시를 잘 못 내렸기 때문일까."

**답**: Claude Opus 모델 탓이 아님. **지시 불완전(60%) + 검증 프로세스 결함(40%)의 구조적 문제**. Sprint 1 백로그에 "UI는 Sprint 3" 명시 → Sprint 3 계획 수립 시 태스크 누락이 발단. 추적성 매트릭스에 "UI 컬럼" 자체가 없어 엔진 검증만으로 "PASS"로 기록된 것이 2차 원인. 재발 방지는 **3단계(Engine/UI/E2E) 매트릭스 의무화 + 실행 가능한 Playtest 시나리오 강제 + 백로그 이월 프로세스 공식화**가 핵심.

### 즉시 실행 예정
본 감사 보고 작성 직후 다음 순서로 실행:
1. **P0** — 재배치 합병 UI 복원 + 브라우저 검증 (20분)
2. **P1** — 하이라이트+히스토리 커밋 + 타이머 max 600 + E2E 합병 테스트 1건 (50분)
3. **P2** — 재배치 나머지 3유형 + 추적성 매트릭스 4열 확장 (Sprint 6 후반)
4. **P3** — Playtest 결정론적 전환 + 규칙 체크리스트 템플릿 (Sprint 7)

---

**본 문서의 의의**: 특정 UX 버그(재배치 합병 미구현) 수정이 아니라 **같은 종류의 누락이 또 발생하지 않도록 하는 프로세스 개선안**이다. 향후 다른 기능에서 유사한 빈틈이 발견될 경우 본 보고서의 3단계 매트릭스 방법론을 재사용할 것을 권장한다.
